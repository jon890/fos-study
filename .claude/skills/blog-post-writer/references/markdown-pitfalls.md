마크다운 렌더 함정과 cross-link 규칙. 글 작성 직후 자가점검에 사용한다. `<file>` 은 작성한 글 경로.

## 작성 직후 통합 자가점검 체크리스트

글을 저장한 직후 아래를 **순서대로 전부** 실행한다. 하나도 건너뛰지 않는다.

스크립트 점검:

```bash
grep -nE '\*\*"|"\*\*' <file>      # L. bold 안 따옴표 → 따옴표 제거
grep -nE '[┌│▼└┐┘─├]' <file>      # M. ASCII 박스 다이어그램 → mermaid (트리 제외)
grep -n "~" <file>                 # G. paragraph당 ~ 2개+ → 취소선 함정, 이스케이프
grep -n "§" <file>                 # H. § → 평문 치환
grep -nE "^#+ [0-9]+\." <file>     # K. heading 숫자 prefix 금지
grep -nE "(위 [0-9]+|[0-9]+개 항목|[0-9]+번 항목|섹션 [0-9]+)" <file>  # K. 본문 숫자 cross-ref
```

수동 점검 (스크립트로 안 잡힘):

- **I. 영문 직역 의심 단어** — 영문 그대로 쓴 용어 중 한글이 더 익숙해 보이는 게 있으면 웹 검색으로 짧게 확인 후 한글 우선 + 영문 병기. 의심 가는 것만.
- **J. 한 단락 정보 4개+ 또는 콤마 3개+ 나열** → 리스트로 변환
- **명사형 종결 → 동사 종결** (본문 평문 문장)
- **A. 인사이트 blockquote 2~3개 이하** 인지 확인
- **cross-link 대상 파일 존재**(`ls`)·**H1 제목 표시텍스트** 사용 여부 확인

## 하위 문서 링크 패턴

특정 기술 개념을 섹션으로 다루고, `fos-study` 안에 해당 개념의 상세 문서가 있다면 섹션 제목 옆에 링크를 달아준다.

```markdown
## @StepScope 빈 충돌 문제 ([Spring Batch @StepScope 정리](../../java/spring-batch/step-scope.md))
```

링크 걸기 전에 해당 파일이 실제로 존재하는지 확인:
```bash
ls /Users/nhn/personal/fos-study/java/spring-batch/
```

없는 파일에 링크를 걸지 않는다. 상세 문서가 없으면 링크 없이 섹션 제목만 쓴다.

폴더 인덱스(`README.md`)를 가리키는 링크는 글이 아니라 `/category/<폴더경로>` 카테고리 페이지로 라우팅된다.
폴더 전반을 가리킬 때는 README 를, 특정 개념이면 그 개념의 개별 글을 가리킨다.

### 10-A. 표시 텍스트는 H1 제목 — 백틱 경로 금지

자동 변환 / 무심결에 만들어지는 `[\`path.md\`](path.md)` 패턴은 fos-blog 렌더링 시 백틱과 경로가 그대로 노출돼 부자연스럽다. **반드시 대상 문서의 H1 제목을 표시 텍스트로** 사용한다.

```markdown
# ❌ 잘못 — 경로가 본문에 노출
관련 개념은 [`connection-pool.md`](./connection-pool.md) 참고.

# ✅ 올바름 — H1 제목으로
관련 개념은 [커넥션 풀 크기](./connection-pool.md) 참고.
```

H1 추출 + 정리 규칙 (긴 H1 은 짧게):

```bash
# 대상 문서의 H1 추출
grep -m1 '^# ' /Users/nhn/personal/fos-study/database/connection-pool.md | sed 's/^# //'
# → "커넥션 풀 크기는 얼마나 조정해야 할까?"
```

- `^\[초안\]\s*` prefix 제거
- ` — `, `: `, ` (` 중 첫 매치 앞부분만 사용 (긴 부제 제거)
- 그래도 40자 초과 시 `?` / `.` 위치에서 자르기

### 10-B. 앵커 link — 특정 섹션 깊은 link

다른 문서의 특정 섹션이 진입점이라면 `path.md#섹션-앵커` 형태로 깊은 link 를 박는다. fos-blog 의 `resolveMarkdownLink()` 가 앵커를 그대로 보존한다.

```markdown
# 같은 문서 내 섹션
백분위수 정의는 [Latency 백분위수](#latency-백분위수--p50-p95-p99-p999-가-무엇인가) 참고.

# 다른 문서 특정 섹션
백분위수 표현이 익숙하지 않으면
[Observability 입문](../architecture/observability-basics.md#latency-백분위수--p50-p95-p99-p999-가-무엇인가)
의 "Latency 백분위수" 섹션 먼저.
```

앵커 생성 규칙 (GFM kebab-case):

- 헤딩 텍스트 → 소문자 변환
- 공백 → `-`
- 특수문자 (`?`, `,`, `.`, `:`, `(`, `)` 등) 제거
- 연속된 `-` 는 그대로 유지 (예: ` — ` → `--`)
- 한글은 그대로 유지

**렌더러 차이 주의** — anchor 변환 규칙은 렌더러마다 미세하게 다르다. 박은 후 fos-blog 에서 실제로 점프되는지 한 번 확인 권장. 안 되면 "{문서}의 \"{섹션}\" 섹션" 형태 본문 참조 + 문서 link 만 박는 fallback.

### 10-C. 이탤릭+괄호 강조는 bold+괄호 분리로

`*한글(영문)*` 같은 이탤릭+괄호 조합은 GFM/fos-blog 파서가 단어 경계 인식 못 해 강조가 렌더링 안 된다. **반드시 bold + 괄호 분리** 형태로 작성.

```markdown
# ❌ 잘못 — 이탤릭이 안 걸림
운영에서 어디에 *격벽(bulkhead)*을 세우는가
Union-Find 는 정확히 *연결성(connectivity)*만을 다룬다

# ✅ 올바름 — 프로젝트 마크다운 Bold+괄호 룰 일관
운영에서 어디에 **격벽**(bulkhead)을 세우는가
Union-Find 는 정확히 **연결성**(connectivity)만을 다룬다
```

프로젝트 마크다운 "Bold + 괄호 패턴" 룰의 자연스러운 확장.

## Cross-link 후보 발굴 (작성 직전 자동 실행)

본문 작성 직전에 글의 핵심 키워드로 fos-study 전역을 ripgrep 해 link 후보를 자동 발굴한다. 작성자가 "이 글에서 다른 어디로 link 걸지" 를 머릿속으로만 떠올리면 빠뜨리기 쉽다.

**절차** (writer 가 자동 실행):

1. **키워드 추출** — 글의 H1·H2 / 주요 강조점에서 5\~10개 키워드 골라 배열로
2. **전역 grep**:
   ```bash
   KEYWORDS=("Outbox" "HikariCP" "Circuit Breaker" "p99 latency" "Bulkhead")
   for kw in "${KEYWORDS[@]}"; do
     echo "== $kw =="
     rg -l --type md "$kw" /Users/nhn/personal/fos-study/ \
       | grep -v "/node_modules/" | grep -v "$(basename {현재 글 경로})" | head -5
   done
   ```
3. **후보 평가** — 각 매치 파일의 H1 추출 → 본문 흐름상 자연스러운 자리 1\~2건만 선정
4. **링크 박기** — 표시 텍스트는 H1 제목 (10-A 규칙), 깊은 link 면 앵커 (10-B)
5. **작성 직후 자가 audit** — `docs-audit` 의 축 4 (cross-link) 가 추가 후보를 보고하면 사용자 동의 후 보강

**작성 원칙**:

- 한 paragraph 에 link 3개 넘으면 가독성 떨어짐 — 가장 가까운 1\~2개만 박고 나머지는 별도 줄
- 자기소개·도입부에는 link 자제 — 본문 흐름이 깨진다
- "관련 / 참고" 섹션에는 link 모아두기 OK

## 글 품질 안티패턴 — 자주 빠지는 함정

글을 쓸 때 형식적으로는 가이드를 지켰어도 **결과적으로 AI 티가 나거나 매끄럽지 않게** 되는 패턴들이 있다. 아래는 실제 작성 후 비판적으로 본 결과 발견된 함정이다.

### A. 인사이트 blockquote 과다

`> **인사이트**` 블록을 각 섹션마다 박으면 "AI가 배치한 하이라이트"로 읽힌다.

- **상한 기준**: 한 글에 **2~3개 이하**. 이걸 넘으면 정말 필요한 것만 남기고 나머지는 본문에 풀어서 녹인다.
- **블록으로 뽑는 기준**: 그 섹션에서 **다른 한 줄로 요약 불가능한 유일한 결론**일 때만. 본문에서 이미 설명한 내용을 한 번 더 박스로 치지 않는다.
- 블록 대신 본문 마지막 문장에 한 줄로 마무리하는 편이 대부분 더 자연스럽다.

### B. 회고/협업 섹션 박제

`## 내 기여 범위` / `## 협업 흐름` / `## 지금 보면` 3종 섹션 세트를 기계적으로 박으면 **이력서 bullet point** 같은 톤이 된다. writing-style 의 회고와 협업 항목에서 상세히 다뤘지만 다시 강조:

- 섹션 제목은 **이야기를 이끄는 제목**으로 (`## 협업이 이 기능을 만들었다`, `## 미션 수 상한이 설계를 결정했다`)
- 정보만 나열하지 말고 **어느 팀의 어떤 질문·요구가 설계를 바꿨는지** 인과를 드러냄
- "지금 보면"은 **라이브러리 업그레이드형 회고**(KMS, 신버전, Caffeine 등)만 나열하면 피상적. **설계 가정·비낙관 경로·전제 추적** 같은 메타 관점을 최소 하나 포함

### C. 의사코드 주석 남용

`// 개념 설명용 의사코드` 한 줄 달아놓고 실제 내용은 실코드와 동일한 패턴. L2 정책의 **형식만 갖추고 실질은 빠뜨린** 상태다.

- 정말 의사코드라면 **축약된 흐름**(4~5줄)만. 함수 바디 전체 인용은 금지.
- 실코드에 가깝게 유지할 거라면 L1(일반 패턴) 수준임을 확신한 뒤 **주석 없이** 둔다.
- 결정 기준: **"이 코드 블록을 이직 포트폴리오로 제출할 수 있는가?"** No면 더 축약한다.

### D. 중복 인사이트 — 허브 지정

한 프로젝트 내에서 여러 글을 쓰면 같은 인사이트(예: 락/리로드/캐시 패턴)가 반복된다. 3글에서 같은 이야기를 조금씩 다르게 변주하면 독자가 연결해 읽을 때 **복사-붙여넣기 느낌**이 난다.

- **허브 글 하나를 지정**하고 거기에 상세를 모음
- 다른 글에서는 간단히 요약 후 `[허브 글 참고](./hub.md)` 링크로 처리
- 허브 글 이외에선 같은 인용구·같은 코드 블록을 반복하지 않는다

### E. 자기 PR 톤 (문체 가이드 상세)

"내가 기반을 만들었다"형 표현은 의도 중심("다른 사람이 쓰기 편하게 API를 좁히는 감각이 생겼다")으로 치환.

### F. 전면 재작성 주의

기존에 좋은 문장이 들어 있는 문서를 고도화할 때 전면 재작성하면 **무의식적으로 축약한 부분이 생긴다**. 실제로 이 스킬 업데이트 과정에서도 원본의 좋은 문장 몇 개를 누락한 사고가 났다.

- **기본 동작은 섹션 단위 수정**
- 전면 재작성할 때는 **원본 vs 새 버전 diff를 반드시 훑는 단계**를 거침
- "이 문장이 원본에 있었는데 왜 사라졌나"를 체크리스트로 돌리면 사고가 준다

### G. `~` 취소선 함정

GitHub Flavored Markdown은 한 paragraph 안에 `~`가 두 번 이상 등장하면 두 `~` 사이를 **취소선(`<del>`)**으로 렌더링한다. 범위 표기를 자주 쓰는 기술 글에서 무심코 발동되기 쉽다.

위험한 패턴:
- `60~100건` 한 문장에 두 번 이상: `60~100건, 30분~1시간 사이` → "0건, 30분"이 취소선으로 묶일 수 있음
- 같은 paragraph에 `2026.04 ~ 2026.05`와 `1~2분`이 같이 등장
- `1~2분 지속됐다. ... 15~30초` 한 문단

대응:
- **백슬래시 이스케이프**: `60\~100건`, `1\~2분` — 가장 안전
- **표현 변경**: "60부터 100건", "1분에서 2분", `2026.04–2026.05` (en dash)
- **코드 블록 보호**: `` `60~100` `` 안에 넣으면 파서가 건너뜀

자가 검증 절차:
1. 글 작성 직후 `grep -n "~" <file>` 또는 에디터 검색으로 모든 `~` 위치 나열
2. 한 paragraph(빈 줄로 분리되는 단위) 안에 `~`가 **2개 이상**이면 위험 — 이스케이프 또는 표현 변경
3. 헤더 메타 줄(`**진행 기간**: 2026.04 ~ 2026.05`)은 단독이라 보통 안전하지만, 같은 헤더 블록에 다른 `~`가 있으면 점검

코드 블록 안의 `~`는 발동하지 않으니 카운트에서 제외한다 (`grep` 결과를 볼 때 ``` 라인 사이는 무시).

### H. `§` 특수문자 금지

섹션 참조를 표기할 때 `§3`, `§14-G` 같이 section sign(`§`)을 쓰지 않는다. 본문에서도, 스킬·가이드·내부 문서에서도 일관되게 금지.

이유:
- 한국어/영어 문서에서 익숙하지 않은 기호라 가독성이 떨어진다
- 일부 폰트·터미널·뷰어에서 깨지거나 다른 글자로 보일 수 있다
- 평문 숫자/한국어 표현이 더 명확하다

대안:
- ❌ `§3 참조`, `§14-G에 상세`
- ✅ `3번 항목 참조`, `14번 항목 G에 상세`, `위 3번 섹션`, `(섹션 3)`

자가 점검: `grep -n "§" <file>` 으로 잔존 여부 확인.

### I. 영문 용어 직역 — 한국 기술 글에서 쓰는 표현 우선

영어 공식 문서·스펙·라이브러리에서 정식 용어로 쓰이는 단어를 그대로 직역하면 한국 독자가 한 번 더 멈춰서 의미를 풀어야 한다. 기술적으로는 정확해도 가독성이 떨어진다.

실제 사례 (envelope):

- ❌ "사내 표준 응답 envelope을 쓰고 있어서"
- ✅ "**공통 응답 포맷**(response envelope)을 쓰고 있어서"

영어권에선 `response envelope`이 JSend·JSON:API 등의 정식 용어지만, 한국 블로그에선 "공통 응답 포맷" 또는 "공통 응답 형식/객체"가 압도적이고 envelope 직역은 거의 없다.

판단 절차:

1. 영문 그대로 쓴 용어가 보이면 한국 기술 블로그에서 자주 쓰는 표현이 따로 있는지 **의심**한다 (모든 단어를 검색하지 않는다)
2. 의심되면 웹 검색으로 짧게 검증 — `"<용어> spring boot 블로그"` 정도면 충분
3. 한국 표현이 더 익숙하면: **한글 용어를 앞세우고 첫 등장 시에만 영문 병기** (`공통 응답 포맷(response envelope)`)
4. 한국에서도 영문이 표준이면 (`idempotency`, `scrape`, `monotonic`, `graceful shutdown`, `circuit breaker` 등 한국 개발자도 영문으로 더 익숙한 용어) 영문 그대로 두되 처음 등장 시 한 줄로 풀어준다. **억지로 한글화하지 않는다**.

흔한 판단 사례:

| 용어 | 권장 표기 | 비고 |
|---|---|---|
| response envelope | "공통 응답 포맷(response envelope)" | 한국에선 한글이 압도적 |
| back-pressure | "백프레셔(back-pressure)" 또는 "역압" | 분야에 따라 다름 |
| graceful shutdown | "graceful shutdown" | 한국에서도 영문 관용 |
| idempotency | "idempotency(멱등성)" 또는 그대로 | 영문/한글 모두 흔함 |
| fan-out | "팬아웃(fan-out)" | 음차가 일반적 |
| scrape (Prometheus) | "scrape" 그대로 | 영문이 표준 |

원칙: **"이 단어를 한국어로 검색했을 때 한국 개발자 블로그에서 어떤 표기가 가장 많이 보이는가"**가 기준이다. 절대적인 한글화도, 절대적인 영문 보존도 정답이 아니다.

자가 점검: `grep`만으로는 잡히지 않는다. 글을 다시 읽으면서 영문 그대로 등장하는 용어 중 **"이건 한글로도 흔하지 않나?"** 싶은 게 있으면 그것만 검색해서 확인.

### J. 가시성·스캔 가능성 — 줄글로 흩어 쓰지 말 것

정보 밀도가 높아도 시각적으로 흩어져 있으면 독자는 못 읽는다. 다음 신호가 보이면 분해한다.

- **항목이 3개 이상이면 콤마 줄글이 아니라 리스트** — `A + B + C + D 시간 예산 설계로 ...` 같이 4개 컴포넌트를 콤마로 잇지 않는다. bullet 4개로 분리한다.
- **링크 제목은 짧게, 본문 설명은 다음 줄로 분리** — `[긴 제목 — 부연 설명](path) — 또 긴 줄글` 패턴 금지. 제목 자체에서는 식별만 되게 짧게, 부연 설명은 별도 줄/단락에서.
- **사례 누적이 예상되는 섹션은 H3 또는 표 구조로 시작** — "실제 적용 사례", "관련 사례", "추가 적용 케이스" 같은 섹션은 사례가 1건이어도 미리 `### 사례명` 또는 표 헤더 구조로 잡아둔다. 누적 시 자연스럽게 확장된다.
- **본문과 메타 설명 분리** — "이 글의 X 단락이 정확히 그 환경" 같은 cross-reference 메모는 본문 줄에 묻지 말고 별도 줄로 띄운다.
- **단락 안에 정보가 4개 이상 묶이면 분해** — 한 paragraph 가 정보 항목을 4개 이상 담고 있으면 사람 눈이 흐른다. 헤더·리스트·표 중 하나로 쪼갠다.

나쁜 예:

```
## 실제 적용 사례

- [OCR 서버 배포·스케일인 시 503 에러 수정 — Graceful Shutdown 미적용](path) — NHN Cloud Container 30초 고정 예산 안에서 Envoy `drain_listeners` + preStop sleep + Python gRPC `server.stop(grace=12)` + supervisord `stopwaitsecs=17` 시간 예산 설계로 30~60초 503 묶음을 해결한 사례. 이 글의 "Kubernetes + NCS 환경에서의 시간 예산" 단락이 정확히 그 환경이다.
```

좋은 예:

```
## 실제 적용 사례

### NHN Cloud Container 30초 고정 예산 하 OCR gRPC 서버 503 해결

이 글의 "Kubernetes + NCS 환경에서의 시간 예산" 단락이 정확히 그 환경이다.

- preStop 스크립트에서 Envoy `drain_listeners` 호출
- preStop sleep 15s 로 Endpoints 전파 대기
- Python gRPC `server.stop(grace=12)` 로 in-flight RPC 처리
- supervisord `stopwaitsecs=17` 로 grace + 여유 확보

→ [OCR 서버 배포·스케일인 시 503 에러 수정](path)
```

자가 점검: 글 작성 후 다시 읽으며 **"한 줄에 콤마가 3개 이상이면서 정보 항목이 콤마로 분리된 paragraph"** 가 있는지 본다. 있으면 리스트로 변환.

### K. Heading 에 숫자 prefix 금지 (블로그 글 한정)

본 규칙은 **fos-blog 등 자동 번호 매김 시스템에 sync 되는 문서에만** 적용된다. skill / docs / ADR / 코드베이스 내부 markdown 같이 자동 번호 매김 시스템 밖의 문서에는 적용하지 않는다 — 오히려 에이전트가 본문 cross-ref ("1단계 진행", "2단계 선택") 할 때 명시적 숫자가 식별을 쉽게 한다.

블로그 글에서는 `## 1. 소제목`, `## 0. 들여쓰기` 처럼 heading 앞에 숫자 + 마침표 prefix 를 붙이지 않는다. fos-blog 렌더링이 자동으로 번호를 매기기 때문에 원본에 번호를 박으면 **렌더 시 "1. 1. 소제목"** 처럼 이중 번호가 나오거나 자동 번호와 어긋난다.

- ❌ `## 1. 변수와 타입`, `### 0. 들여쓰기가 문법이다`
- ✅ `## 변수와 타입`, `### 들여쓰기가 문법이다`

본문에서 "위 7번 항목 참조", "12개 항목을 알면" 같이 **숫자로 cross-ref 한 경우도 함께 점검**한다. 자동 번호와 어긋날 수 있으니 "위 항목들을 알면", "여기까지만 알면" 처럼 숫자를 빼고 문맥으로 가리킨다.

기존 vault 안에 숫자 prefix 가 박힌 글이 있다 해도 (예: `multimodal.md`) 이는 과거 관행이고 신규 작성 시에는 적용하지 않는다.

자가 점검: `grep -nE "^#+ [0-9]+\." <file>` 으로 한 줄도 안 나오는지 확인. 같이 본문에서 `grep -nE "(위 [0-9]+|[0-9]+개 항목|[0-9]+번 항목|섹션 [0-9]+)"` 으로 숫자 cross-ref 잔존 확인.

### L. bold 안 따옴표 — `**"..."**` 파싱 실패

`**"텍스트"**` 처럼 bold(`**`) 안에 큰따옴표가 바로 붙으면, 일부 마크다운/fos-blog 파서가 따옴표를 단어 경계로 못 읽어 **bold 가 아예 안 걸린다**. 강조하려고 박은 게 평문으로 렌더된다.

- ❌ `Ingress는 **"어느 경로를 어느 Service로"**를 적은 규칙이다`
- ✅ 따옴표를 빼고 bold 만: `Ingress는 **어느 경로를 어느 Service로**를 적은 규칙이다`
- ✅ 또는 따옴표를 bold 밖으로: `Ingress는 "**어느 경로를 어느 Service로**"를 적은 규칙이다`

기본은 **따옴표 제거 + bold 유지**. 인용 뉘앙스가 꼭 필요하면 따옴표를 밖으로 뺀다.

자가 점검: `grep -nE '\*\*"|"\*\*' <file>` 으로 잔존 확인. bold 안에 따옴표가 여러 쌍 있는 경우(`**"A"와 "B"**`)는 중간 따옴표까지 함께 제거한다.

### M. 다이어그램은 ASCII 박스 대신 mermaid

fos-blog 는 mermaid 를 렌더한다(`flowchart`, `sequenceDiagram` 등). 관계도·흐름도·시퀀스를 `┌─│▼└` ASCII 박스로 그리면 정렬이 깨지고 수정도 번거롭다. **mermaid 코드블록으로 그린다.**

- 관계·트래픽 흐름 → `flowchart TB`(위→아래) 또는 `LR`(좌→우). `subgraph` 로 경계(namespace 등) 표현.
- 시간 순 상호작용(handshake, 요청·응답) → `sequenceDiagram`.
- 노드 라벨 줄바꿈은 `<br/>`, 화살표 설명은 `-->|"설명"|`.
- **예외 — 디렉터리/파일 트리는 코드블록 유지**(mermaid 부적합). `├──`/`└──` 트리는 그대로 둔다.

자가 점검: `grep -nE '[┌│▼└┐┘─├]' <file>` 으로 ASCII 박스 다이어그램 잔존 확인(트리 제외). 있으면 mermaid 로 전환.
