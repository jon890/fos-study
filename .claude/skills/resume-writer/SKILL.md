# 이력서 작성 스킬

## The Insight

좋은 이력서는 기억에 의존하지 않는다. task/ 폴더의 실제 구현 기록을 읽고, 그 안의 클래스명·메서드명·수치를 그대로 가져와야 면접에서 답할 수 있는 이력서가 된다.

토스가 공개한 이력서 가이드의 핵심:
- **폭넓은 프로젝트보다 구체적으로 만든 기능과 해결한 문제** 중심
- 기술 선택 이유, 대안 기술 비교, 트레이드오프까지 서술
- 숫자로 말하는 임팩트 (58배, 25%, OOM → 20B 등)
- "왜 이 기술이어야만 했나" — 탈락자의 공통 약점

## Recognition Pattern

- 특정 회사/직무에 맞는 이력서를 새로 작성하는 경우
- 기존 이력서에 새 경험을 추가하거나 내용을 강화하는 경우
- 회사별로 다른 강조 포인트가 필요한 경우
- 이미 분석 문서(interview/ 폴더)가 있으면 그것을 먼저 활용

## The Approach

### 0. 입력 확인

다음 정보를 확인한다. 없으면 먼저 요청한다:
- 지원 회사명 / 직무
- 개인 정보 (이름, 이메일, 전화번호, GitHub)
- 재직 회사명, 총 경력 기간
- 기존 이력서 파일 경로 (있는 경우)
- 강조하고 싶은 경험이나 제외할 내용

### 1. 직무 분석 문서 확인

`interview/` 폴더에 해당 회사 분석 문서가 있는지 확인한다.

```
interview/{회사명}-{직군}.md
```

있으면 다음을 파악한다:
- 공고의 핵심 요구사항과 우대사항
- 기술 스택
- 이미 파악된 경험 매핑 테이블
- 상대적 약점 (덜 강조할 부분)

없으면 job-analyzer 스킬 실행을 먼저 제안한다.

### 2. task/ 폴더 탐색

각 팀 README.md를 먼저 읽어 전체 구조를 파악한 후, 공고와 관련도 높은 문서를 상세히 읽는다.

```
task/
├── {팀명}/README.md     ← 먼저 읽기
└── {팀명}/{문서}.md     ← 관련도 높은 것 상세 읽기
```

탐색 시 수집할 내용:
- **클래스명·메서드명**: 면접에서 꺼낼 수 있는 구체적 이름
- **수치**: 배수, %, ms, GB 등 측정 가능한 임팩트
- **기술 선택 이유**: "왜 A가 아닌 B인가" 논리
- **문제 → 원인 → 해결** 서사 구조
- **팀에 제안하거나 자발적으로 추진한 작업** (오너십 어필)

### 3. 이력서 구조 설계

공고 요건과 실제 경험을 교차 분석해 다음을 결정한다:

- **경험 섹션 순서**: 최신순 (역시간순)
- **강조할 회사/팀**: 공고와 가장 관련 높은 경험 먼저
- **제외할 내용**: 공고와 무관하거나 어필이 약한 항목
- **Key Projects 구성**: 면접에서 깊게 파고들 수 있는 2~3개

### 4. 이력서 작성 원칙

#### Bullet 작성 규칙

```
❌ "Redis를 활용해 캐시 성능을 개선했습니다"
✅ "MQ Fanout 구조로 다중 인스턴스 캐시 정합성 해결 — RabbitMQ/Azure Service Bus 이중화를 DataPublisher 인터페이스로 추상화해 환경 무관하게 동일 동작 보장"
```

- **문제 → 해결 → 임팩트** 순서
- 기술 선택 이유를 한 문장으로 포함
- 클래스명/메서드명은 backtick으로 표기
- 수치가 있으면 반드시 포함
- "직접", "자발적으로" 같은 표현은 남용하지 않음 — 오글거림 주의

#### 강조 레벨 조절

- 최근 회사 + 공고 관련도 높은 경험: 상세 bullet + Key Projects
- 오래된 회사 / 관련도 낮은 경험: 간략 bullet, 강조 태그 없음
- 기술 스택이 맞지 않는 회사 (초기): 담백하게 사실만

#### Summary 작성

- 도메인 나열 (게임/베팅/거래소/AI 등)
- 자신있는 강점 1~2개 (추상화, 테스트 인프라 등)
- 현재 하고 있는 일로 마무리

### 5. HTML + MD 동시 작성

두 포맷을 함께 작성한다.

#### HTML 파일 요건

```html
<!-- PDF 출력을 위한 필수 설정 -->
@page { size: A4; margin: 18mm 20mm; }
-webkit-print-color-adjust: exact;
print-color-adjust: exact;

<!-- 폰트: 시스템 한국어 폰트 우선 -->
font-family: -apple-system, 'Apple SD Gothic Neo', 'Malgun Gothic', sans-serif;
```

클래스 구조:
- `.job` / `.job-header` / `.job-stack` / `.period` — 경력 항목
- `.project` / `.org` / `.bg` — Key Projects
- `.highlight` — 수치 강조 (노란 배경)
- `.tag` / `.tags` — 기술 스택 뱃지

#### 파일명 규칙

```
resume/{회사명-케밥케이스}-resume-{직군}.html
resume/{회사명-케밥케이스}-resume-{직군}.md
```

예: `resume/toss-securities-resume-product.html`

### 6. 검증

작성 후 다음을 확인한다:

- [ ] 모든 클래스명·메서드명이 task/ 파일과 일치하는가
- [ ] 수치가 기억이 아닌 문서에서 나왔는가
- [ ] "Redis Cluster", "AOF" 등 실제 구현하지 않은 내용이 없는가
- [ ] 각 bullet에 "왜 이 기술인가"가 담겨 있는가
- [ ] 오글거리는 표현(직접, 최초, 단독 등)이 과도하게 사용되지 않았는가

## 실행 단계

1. 입력 정보 확인 (없으면 요청)
2. **[병렬]** interview/ 분석 문서 확인 + task/ 팀별 README 읽기
3. 관련도 높은 task 문서 상세 읽기 (클래스명·수치 확인 포함)
4. 이력서 구조 설계 및 사용자 확인 (선택적)
5. HTML + MD 동시 작성
6. `resume/` 폴더에 저장
7. 검증 체크리스트 확인
8. 다음 내용을 포함한 실행 요약(`transcript.txt`)을 이력서와 같은 폴더에 저장:
   - 읽은 파일 목록 (task/ 문서, 기존 이력서 등)
   - 기존 이력서를 Read했는지 여부 (update 케이스)
   - 클래스명·수치 출처 검증 결과
9. 저장 경로 + 핵심 구성 요약 전달

## 주의사항

- **코드 검증 우선**: task 문서에 있어도 실제 코드와 다를 수 있음. 확신이 없으면 코드를 직접 확인
- **사용자 확인**: 수치나 임팩트가 불확실한 경우 추측하지 말고 사용자에게 질문
- **과장 금지**: 경험이 없는 기술(Kafka, Kubernetes 등)은 유사 경험으로 솔직하게 대응
- **기존 파일 보존**: 수정 전 반드시 Read로 현재 내용 확인
