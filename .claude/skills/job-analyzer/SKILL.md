# 채용 공고 직무 분석 문서 생성

## The Insight

채용 공고만 읽는 것과 회사의 기술 블로그·실제 사례까지 조사한 뒤 본인 경력과 매핑한 것 사이에는 큰 차이가 있다. 이 스킬은 그 과정을 체계적으로 수행한다:

1. **공고 원문 크롤링** — 직무명, 담당업무, 자격요건, 기술스택을 구조적으로 추출
2. **회사 기술 자료 조사** — 기술 블로그, 공식 자료에서 실제 사용 기술·아키텍처 패턴 파악
3. **지원자 경력 매핑** — task/ 폴더 탐색으로 실제 경험과 공고 요건을 교차 분석
4. **분석 문서 저장** — interview/{파일명}.md 로 저장하고, README.md 업데이트

## Recognition Pattern

- 채용 공고 URL과 함께 분석 요청이 오는 경우
- "이 회사 지원해보려고", "공고 분석해줘", "서류 준비 도와줘" 등
- URL 없이 회사명+직무만 말하는 경우 → URL을 먼저 요청
- 여러 직무를 동시에 분석하는 경우 → 각각 별도 파일 생성 후 비교 분석 추가

## The Approach

### 0. 입력 확인

공고 URL이 있으면 바로 시작. 없으면 먼저 URL을 요청한다.
여러 공고를 동시에 분석할 경우 각 단계를 병렬로 실행한다.

### 1. 공고 크롤링

`WebFetch`로 공고 URL을 가져온다. 추출 대상:

- 회사명, 팀명, 직군, 고용형태, 근무지
- 팀 소개 (미션, 규모, 문화)
- 담당 업무 목록
- 필수/우대 자격요건
- 기술 스택 (언어, 프레임워크, 인프라, 데이터, 모니터링)
- 전형 절차
- 이력서 작성 가이드나 특이사항

페이지 로드 실패 시 사용자에게 공고 내용을 직접 붙여넣어 달라고 요청.

### 2. 회사 기술 자료 조사

공고에서 파악한 회사명을 기반으로 `WebSearch`로 기술 자료를 찾는다.

검색 키워드 조합 (우선순위 순):
```
"{회사명} 기술 블로그"
"{회사명} tech blog"
"{회사명} engineering blog"
"{회사명} backend architecture"
"{회사명} 개발 문화"
```

찾은 블로그/자료에서 `WebFetch`로 **2~3개**의 글을 읽는다. 집중할 내용:
- 실제 사용 기술 스택 (공고에 없는 것 포함)
- 아키텍처 패턴 (MSA, Event-Driven, 캐싱 전략 등)
- 트러블슈팅 사례 (어떤 문제를 어떻게 해결했는지)
- 서비스 규모 (트래픽, 데이터량, 팀 규모)
- 개발 문화 및 코드 리뷰 방식

기술 블로그를 찾지 못하면 뉴스, 채용 다른 공고, GitHub 등에서 보완.

### 3. 지원자 경력 파악

`task/` 폴더를 탐색해 지원자의 실제 경험을 파악한다.

```
/Users/nhn/personal/fos-study/task/
├── {팀명}/README.md    ← 팀별 업무 요약 먼저 읽기
└── {팀명}/{문서}.md    ← 핵심 구현 문서 상세 읽기
```

파악할 내용:
- 사용 기술 스택 (언어, 프레임워크, DB, 캐시, 메시징)
- 구현한 주요 기능 및 시스템
- 성능 최적화, 트러블슈팅 경험
- 아키텍처 설계 경험 (추상화, 공통화, 표준화)
- 규모 관련 경험 (트래픽, 데이터량, 동시성)

### 4. 분석 문서 작성

아래 구조로 마크다운 문서를 작성한다. `interview/cj-oliveyoung-wellness-backend.md`가 참고 기준.

#### 작성 원칙

- **기술 스택 섹션**: 공고 원문의 것 + 블로그에서 파악한 것을 명확히 구분해서 표에 병기
- **내 경험 매핑 테이블**: "Redis 경험 있음" 같은 단순 매핑 금지. "Ehcache + MQ Fanout 기반 멀티 서버 캐시 동기화 설계"처럼 구체적 구현 내용으로
- **약점 분석**: 갭을 숨기지 말고 솔직하게. 대신 유사 경험으로 보완 가능한 방향 제시
- **여러 직무 비교**: 동일 회사 두 직무를 분석할 경우 Platform 문서에 비교표 추가 (toss-securities 패턴 참고)

### 5. 파일 저장 및 README 업데이트

**파일명 규칙**: `{회사명-케밥케이스}-{직군-케밥케이스}.md`
- 예: `toss-securities-server-platform.md`
- 예: `kakao-backend-engineer.md`

저장 경로: `/Users/nhn/personal/fos-study/interview/{파일명}.md`

저장 후 `interview/README.md`가 있으면 새 항목을 추가한다.

## 실행 단계

1. 공고 URL 확인 (없으면 요청)
2. **[병렬]** WebFetch로 공고 크롤링 + WebSearch로 회사 기술 블로그 탐색
3. **[병렬]** 기술 블로그 글 2~3개 WebFetch + task/ 폴더 탐색
4. 분석 문서 작성
5. `interview/{파일명}.md` 저장
6. `interview/README.md` 업데이트 (있을 경우)
7. 사용자에게 저장 경로 + 핵심 분석 요약 (어느 직무가 더 어울리는지 포함) 전달
