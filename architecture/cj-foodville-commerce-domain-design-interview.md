# [초안] CJ푸드빌 디지털 채널 면접: 슬롯 도메인 경험을 커머스 도메인 설계 능력으로 번역하기

## 왜 이 주제가 중요한가

CJ푸드빌 디지털 채널 백엔드 면접에 슬롯/베팅/AI 배치 같은 이력으로 들어가면 가장 먼저 받는 질문이 정해져 있다. "그건 좋은 경험인데, 저희 도메인은 주문·결제·쿠폰·매장 운영이거든요. 그쪽 경험 없이 4년차+ 시니어로 들어와서 도메인 설계 할 수 있나요?" 이 질문에 슬롯 RTP나 RCC 같은 단어를 그대로 던지면 면접관 머릿속에 그림이 안 그려진다. 반대로 "예전에 슬롯도 해봤고 커머스도 비슷할 것 같습니다" 식의 일반론으로 도망가면 도메인 깊이 점수를 못 받는다.

이 문서는 그 사이를 정확히 메우기 위한 답변 playbook이다. 후보자가 NSC 슬롯팀과 SB 베팅팀, AI 서비스팀에서 실제로 다룬 추상화·동시성·이벤트 경계·캐시 정합성·관측성 의사결정을 CJ푸드빌 디지털 채널의 주문/결제/쿠폰/매장 상태 설계 언어로 1:1 번역한다. 단순 키워드 매핑이 아니라, "그 의사결정을 왜 그렇게 했고, 커머스 도메인이라면 같은 사고를 어디에 어떻게 쓸지"까지 같은 호흡으로 답하기 위한 자료다. 도메인 모델링, bounded context, 상태 전이, 불변 조건, 예외 복구, 운영 관측을 슬롯 → 커머스 매핑 표 + 1분 답변 + 꼬리 질문 방어로 구성한다.

관련 개념·세부 도메인은 본 저장소에 이미 정리돼 있으므로 깊이 들어가지 않고 링크로 넘긴다.

- 주문·재고·노출 분리: [커머스 도메인 모델링](./commerce-domain-modeling-order-inventory-display.md)
- 주문 상태머신·멱등성·Outbox: [e-Commerce 주문·결제 도메인 모델링](./ecommerce-order-payment-domain-modeling.md)
- F&B 도메인 전체 그림: [F&B · e-Commerce 디지털 채널 도메인 한 장 정리](./fnb-ecommerce-domain-overview.md)
- 주문 상태 정합성 기본기: [커머스 주문 상태와 데이터 정합성 기본기](./commerce-order-state-consistency-fundamentals.md)
- 결제 멱등성: [결제 도메인 멱등성과 트랜잭션 재시도 기본기](./payment-idempotency-transaction-basics.md)
- 쿠폰 동시성 기본기: [쿠폰/프로모션 동시성과 정합성 기본기](./coupon-promotion-concurrency-basics.md)
- F&B 주문/픽업 상태머신: [F&B 주문/매장/픽업 상태머신 설계](./fnb-order-store-pickup-state-machine.md)

## 핵심 통찰: 도메인 단어는 다르고, 의사결정 축은 같다

면접관이 "커머스 경험 없이 가능하냐"고 물을 때 진짜로 보고 있는 것은 회사 도메인 단어를 외우고 있느냐가 아니다. 다음 6개 축에서 의사결정을 내려본 적이 있냐를 본다.

- 어디까지를 하나의 Aggregate로 묶고, 어디부터 이벤트로 끊을 것인가
- 어떤 상태 전이가 불법인지 enum + 전이 규칙으로 강제할 수 있는가
- 동시 요청이 같은 자원에 부딪힐 때 락/유니크 키/낙관적 락 중 무엇을 고를 수 있는가
- 트랜잭션 커밋 이후 외부 시스템 전파를 어떤 패턴으로 보장할 것인가
- 캐시와 진실 원천이 어긋날 때 어떻게 좁히고 어떻게 관측할 것인가
- 장애가 났을 때 1차 대응에서 무엇을 보고 무엇을 끄고 무엇을 재시도할 것인가

후보자가 슬롯/베팅/AI 배치에서 한 일을 이 6축 위에 다시 줄 세우면, 커머스 도메인의 같은 자리에 그대로 끼워넣을 수 있다. 면접 답변은 이 매핑을 한 번 보여주는 형태로 짠다.

## 매핑 1축: Bounded Context와 Aggregate 경계

슬롯 도메인에서 후보자는 `SlotTemplate` / `BaseSlotService` 추상화를 슬롯 5종이 누적된 뒤에야 도입했다. 처음부터 추상화했으면 잘못된 경계를 그었을 것이라는 판단으로, 반복이 보일 때까지 기다린 결정이다. 같은 사고는 커머스에서 거의 동일한 모양으로 나온다.

| 슬롯 도메인 | 커머스 도메인 | 공통 의사결정 |
|---|---|---|
| `SlotTemplate` / `BaseSlotService` (스핀 흐름 추상) | `OrderCommandService` / `PaymentCommandService` (주문·결제 흐름) | 변하는 부분(slot별 페이라인 vs 주문별 결제수단)만 hook으로 위임 |
| `RccSpinResultAnalyzer` 슬롯별 구현 | 매장별 가용성 정책, 브랜드별 가격 정책 구현 | 인터페이스 + 슬롯/브랜드별 구현 주입으로 if-else 분기 제거 |
| 잭팟 포함 여부 → `NoOpJackpotService` | 픽업/배달 여부 → `NoOpDeliveryService` | 기능 미보유 컨텍스트를 분기로 다루지 않고 무동작 구현체로 다룬다 |
| 슬롯 한 판이 하나의 Aggregate Root | 주문 한 건이 하나의 Aggregate Root | 같은 트랜잭션에 묶일 데이터만 안에 둔다 |
| 슬롯 결과 → 통계/미션/알림은 이벤트 | 주문 확정 → 적립/알림/CRM은 이벤트 | 비핵심 후처리는 같은 트랜잭션 밖으로 끊는다 |

답변 예시(1분, 자기소개 직후 도메인 적합도 질문 대응):

> "슬롯팀에서도 처음에는 슬롯마다 스핀 로직을 따로 짰는데, 5종이 누적되면서 공통 흐름과 슬롯별 변동점이 분리가 됐고 그 시점에 `SlotTemplate` + `BaseSlotService`로 묶었습니다. 같은 사고를 커머스로 옮기면, 주문이라는 같은 흐름 안에 매장 픽업/배달/예약/단체주문이 들어올 때 모든 채널이 한 클래스에 if로 쌓이지 않게 `OrderFulfillmentStrategy` 같은 인터페이스로 끊고, 픽업과 배달이 공통으로 갖는 흐름은 상위에 두는 형태가 될 것 같습니다. 회사 도메인 단어는 처음이지만 어떤 단위가 Aggregate가 되고 어떤 단위가 전략 위임이 되는지 판단해 본 경험은 그대로 가져갈 수 있습니다."

## 매핑 2축: 상태머신과 불변 조건

슬롯 스핀과 RCC 캐시 상태는 enum + 전이 규칙으로 다뤘다. 캐시 충족 판정이 안 되면 일반 스핀, 캐시 히트면 캐시 반환, 캐시 비동기 생성 트리거. 이 흐름은 커머스의 주문 상태머신으로 거의 그대로 옮겨진다.

```text
슬롯:    SPIN_REQUESTED → CACHE_HIT or NORMAL_SPIN
         CACHE_HIT      → RESULT_RETURNED + ASYNC_CACHE_REFILL
         NORMAL_SPIN    → RESULT_COMPUTED → STATS_EVENT

주문:    CREATED         → PAYMENT_PENDING
         PAYMENT_PENDING → PAYMENT_APPROVED | PAYMENT_FAILED | CANCELED_BY_USER
         PAYMENT_APPROVED→ STORE_ACCEPTED | STORE_REJECTED
         STORE_ACCEPTED  → READY_FOR_PICKUP | OUT_FOR_DELIVERY → COMPLETED
         어디서든        → REFUND_REQUESTED → REFUNDED
```

두 도메인 모두에서 면접관이 보고 싶은 것은 같다.

- 상태 컬럼이 자유 문자열이 아니라 enum이라는 점
- 모든 전이가 코드의 한 곳에서 검증된다는 점(전이 표 또는 `canTransitionTo`)
- 불법 전이를 시도하면 도메인 예외가 던져진다는 점
- 같은 상태로의 재진입을 멱등으로 처리한다는 점(중복 요청, 재시도)

후보자의 슬롯 경험에는 캐시 충족 판정에서 동일 조건 중복 생성을 막기 위해 DB 유니크 키 + 예외 처리를 선택한 의사결정이 있다. 커머스에서는 같은 패턴이 결제 멱등성에서 그대로 재현된다. 주문 ID + 결제수단 + 시도 차수 조합을 유니크 키로 잡아 중복 결제 요청을 DB가 막게 하고, 어플리케이션은 unique constraint 위반을 "동일 결제 시도"로 해석해서 기존 결과를 그대로 돌려준다.

답변 예시(꼬리 질문 "그럼 동시성 처리 경험은요?"):

> "슬롯 RCC에서는 캐시 항목 동시 생성 충돌을 막기 위해 분산락 대신 DB 유니크 키 + 예외 처리를 선택했습니다. 빈도가 낮고 충돌 시 비용이 작아서 락보다 단순한 안전장치가 더 운영 친화적이라고 봤습니다. 커머스 결제로 옮기면 idempotency key를 유니크 인덱스로 잡고 두 번째 요청은 같은 결과를 돌려주는 형태가 됩니다. 반면 쿠폰 선착순처럼 충돌 빈도가 높고 즉시 응답이 필요한 케이스라면 Redis 단일 인스턴스 + Lua 스크립트나 RDBMS의 atomic update 한 줄로 차감 → 음수면 실패 식으로 처리할 겁니다. 락 종류는 충돌 빈도와 실패 비용을 보고 고른다는 원칙으로 답변드리고 싶습니다."

## 매핑 3축: 트랜잭션 경계와 이벤트 발행

후보자의 가장 강한 카드 중 하나가 Kafka Transactional Outbox Pattern 운영 경험이다. `@TransactionalEventListener(AFTER_COMMIT)` + `Propagation.REQUIRES_NEW` + 실패 메시지 DB 저장 + 스케줄러 재전송 + traceId 추적까지 한 셋트로 운영했다. 커머스에서 같은 패턴이 들어갈 자리는 명확하다.

- 주문 확정 → 매장 POS 접수 큐 발행
- 결제 승인 → CRM/적립/알림 큐 발행
- 환불 승인 → 재고 환원, 쿠폰 복구 큐 발행
- 멤버십 등급 변경 → 캐시 무효화 fanout

답변 예시(꼬리 질문 "주문 들어오면 매장 POS, 알림, 적립을 같이 처리해야 하는데 트랜잭션 어떻게 잡을까요?"):

> "한 트랜잭션 안에 외부 호출을 같이 묶으면 PG/POS가 느려졌을 때 DB 락 점유 시간이 같이 늘어나서 다른 주문도 같이 막힙니다. 그래서 DB는 주문 + Outbox 테이블 한 트랜잭션에만 쓰고, 메시지 발행은 `AFTER_COMMIT` 리스너에서 합니다. 발행 자체가 실패하면 `REQUIRES_NEW`로 outbox 행 상태를 `FAILED`로 떨어뜨리고 traceId를 같이 적습니다. 스케줄러가 일정 주기로 `FAILED`/`PENDING` 행을 다시 보내고, 매장 POS 쪽은 같은 이벤트를 두 번 받아도 무방하도록 멱등 처리합니다. 슬롯팀에서 이 구조로 운영했었고 커머스에서는 발행 대상이 PG·POS·알림·CRM으로 늘어날 뿐 구조는 같다고 봅니다."

## 매핑 4축: 캐시 정합성 — 다중 서버, 다중 매장

NSC 슬롯팀에서 후보자는 `PostCommitUpdateEventListener` + RabbitMQ Fanout + StampedLock으로 다중 서버 정적 데이터 캐시 정합성을 풀었다. SB 베팅팀에서는 Ehcache + 인메모리 Map 이중 캐시와 MQ Fanout으로 같은 문제를 풀되 RabbitMQ와 Azure Service Bus를 환경별로 추상화 주입했다. 이 경험은 커머스 메뉴/프로모션/매장 운영시간 캐시에 거의 동일하게 들어맞는다.

대표적인 적용 대상은 세 가지다.

- **메뉴/옵션 마스터** — 본사 어드민에서 가격이나 옵션이 바뀔 때 전 매장 서버의 캐시가 한꺼번에 무효화돼야 한다. fanout 패턴이 그대로 들어간다.
- **프로모션 정책** — 시간대 가격, 1+1, 쿠폰 결합 규칙. 운영 중간에 활성 정책이 바뀌면 짧은 지연 안에 모든 서버 인스턴스가 같은 결정을 내려야 한다.
- **매장 운영시간/품절 표시** — 매장 POS에서 들어오는 상태 변경이 디스플레이/주문 가능 여부 캐시에 전파돼야 한다.

답변 예시(꼬리 질문 "어드민에서 메뉴 가격 바꿨는데 일부 매장에서 5분간 옛 가격으로 결제됐다 가정. 어떻게 풀까요?"):

> "원인은 다중 서버 캐시 정합성과 캐시 갱신 중 조회 두 가지가 겹쳤을 가능성이 높습니다. 슬롯에서는 JPA 커밋 이벤트를 받아 RabbitMQ Fanout으로 모든 인스턴스 큐에 같은 메시지를 뿌리고, 각 인스턴스는 받은 시점에 자기 캐시만 선택적으로 갱신했습니다. 갱신 구간은 `StampedLock`의 writeLock으로 잠그고 조회는 `tryReadLock`에 2.5초 타임아웃을 줘서 갱신 중에도 읽기 쪽이 멈추지 않게 했습니다. 커머스 메뉴 가격으로 옮기면 PriceCacheUpdate 이벤트를 fanout으로 흘리고, 각 서버가 가격만 부분 갱신, 그동안 결제 직전에는 캐시가 아니라 DB 가격을 다시 한 번 검증하는 방어 코드를 같이 두는 그림이 될 것 같습니다. 캐시는 노출용, 결제 금액 확정은 DB 진실로 분리하는 게 핵심이라고 생각합니다."

## 매핑 5축: 운영 관측과 장애 1차 대응

AI 서비스팀에서 후보자는 NHN Cloud Container Service의 `terminationGracePeriodSeconds` 30s 고정 제약 아래 preStop sleep 15s + gRPC grace 12s + 여유 3s로 503을 제거했다. 제약을 예산으로 환산하는 사고는 커머스 운영에 그대로 옮긴다.

- 이벤트 오픈 5분 전 트래픽이 평소 10배. 결제 PG TPS 한계가 분명히 있다. 같은 호흡으로 "예산 = PG 승인 TPS × 안전 마진"을 정해두고, 임계 도달 시 대기열·쿨다운·페이지로 흘려 보낸다.
- 매장 POS 연동이 일시 장애. 주문 접수는 받되 POS 발행은 outbox에 쌓고, 일정 시간 초과 시 자동 환불 + 사용자 알림.
- 캐시 미스가 폭증해 DB가 흔들리는 케이스. cache stampede 방어로 single-flight나 mutex 패턴을 미리 둔다.

관측 측면에서도 슬롯에서 traceId를 모든 비동기 메시지에 같이 실어 보낸 경험이 그대로 살아난다. 주문 ID + 결제 ID + 메시지 ID + traceId가 같은 로그 라인에서 검색되는 구조는 장애 1차 대응 5분의 핵심 자산이다.

답변 예시(꼬리 질문 "장애 발생 첫 5분에 무엇을 보세요?"):

> "주문 실패율, 결제 실패율, PG/POS 응답시간, 그리고 traceId 기반 실제 실패 흐름 한 건을 봅니다. AI 서비스에서 graceful shutdown 503을 잡을 때 환경 변수가 강제 30초였는데, 그 안에서 preStop·grace·여유를 합쳐 예산을 짜는 식으로 푼 경험이 있습니다. 커머스에서도 PG TPS, 매장 POS 응답 한계 같은 제약을 먼저 숫자로 잡아두고, 임계를 넘으면 자동으로 결제수단 일부 비활성화·대기열·CS 알림이 켜지는 형태가 운영 첫 5분 시나리오라고 봅니다."

## 자기소개에 박을 슬롯 → 커머스 한 줄

면접 자기소개나 1분 인트로에 그대로 박을 수 있는 문장 세 개. 슬롯/베팅/AI 단어를 최소화하고 의사결정 축으로 번역돼 있다.

1. "운영 트래픽을 받는 자사 서비스 백엔드에서 추상화 시점, 트랜잭션 경계, 캐시 정합성을 직접 결정해 본 경험이 강점입니다."
2. "도메인 단어는 처음이어도 어떤 흐름이 Aggregate가 되고 어떤 후처리는 이벤트로 끊을지 판단해 본 경험이 있어서 주문/결제/쿠폰 같은 자리에 빠르게 자리잡을 수 있다고 봅니다."
3. "PG·POS·CRM 같은 외부 시스템과의 경계는 Outbox Pattern으로 다뤄본 적이 있고, 메뉴/프로모션 캐시 정합성은 fanout 무효화로 다뤄본 적이 있어서 운영형 커머스 백엔드에서 흔히 깨지는 자리에 대한 감각은 가져갈 수 있습니다."

## 압박 질문 방어

면접관이 약점을 캐는 질문을 던질 때 도망가지 않고 사실 기반으로 답한다. 후보자 자가 진단(약점: DB 튜닝 깊이, JPA 드릴, Kafka 운영 디테일, Kotlin)을 기반으로 한다.

- "MyBatis나 JSP 레거시 경험 없이 들어와서 적응 가능한가?" → "Spring Boot 3 + JPA 중심으로 운영해 왔지만, 레거시 영역은 Strangler Fig 패턴 관점에서 새 API를 점진적으로 붙이고 세션·토큰 인증 공존을 분리해 본 적이 있는 동료 코드 리뷰 경험으로 보완하겠습니다. MyBatis도 동적 SQL/resultMap 수준은 빠르게 따라잡을 수 있다고 봅니다."
- "F&B 도메인 모를 텐데 어떻게 빨리 따라잡을 건가?" → "fos-study 저장소에 F&B 도메인 정리를 이미 시작했고, 첫 1\~2주는 회사 도메인 용어집과 실제 주문/결제/쿠폰 흐름 1건을 끝까지 따라가 보면서 매핑하겠습니다. 슬롯도 처음에는 도메인 용어를 따라가는 데 한 달이 들었지만 이후 슬롯 8종을 책임지게 됐습니다."
- "구체 TPS는 얼마였습니까?" → "정확한 측정 결과로만 답을 드리고 싶어서 정성적으로 말씀드리면 ... 식입니다. 슬롯 시뮬레이터에서 JMH로 `ThreadLocalRandom` 70 ops/s 대 `SecureRandom` 1.2 ops/s 차이는 측정해 두었고, 운영 TPS는 매장/시간대 의존성이 커서 측정 단위와 함께 답변드리는 게 정확할 것 같습니다."

## 체크리스트 (면접 전날)

- [ ] 자기소개 30초 안에 "운영 자사 서비스 백엔드 + 추상화·트랜잭션·캐시 의사결정" 키워드가 들어가 있는가

- [ ] 주문 상태머신 / 결제 멱등성 / 쿠폰 동시성 / 캐시 정합성 / Outbox / 운영 1차 대응 다섯 자리에서 각각 1분 답변을 준비했는가

## 마치며

같은 회사를 다녀 본 적이 없다는 사실은 면접에서 약점이 아니라 정직하게 인정하고 넘어가야 할 출발선이다. 후보자의 진짜 카드는 회사 이름이 아니라, "운영 트래픽을 받는 백엔드에서 어떤 의사결정 축 위에 어떤 트레이드오프로 결정을 내렸는가"를 한 흐름으로 설명할 수 있다는 점이다. 슬롯 RCC, Kafka Outbox, 분산 캐시 정합성, graceful shutdown 503 해결은 단어만 다를 뿐 CJ푸드빌 디지털 채널 백엔드가 매일 부딪히는 문제와 같은 자리에 있다. 이 문서가 그 매핑을 면접장에서 1분 안에 꺼낼 수 있는 형태로 굳히는 데 쓰이면 충분하다.
