# Cache-Aside 패턴

**애플리케이션이 캐시와 데이터 저장소를 직접 관리하면서, 요청 시 캐시를 먼저 확인하고 미스 시에만 저장소에서 읽어오는 방식**이다. 이를 **Lazy Loading** 패턴이라고도 부른다.

캐시-저장소 간의 동기화는 애플리케이션의 책임이며, 실제 사용되는 데이터만 선택적으로 캐싱할 수 있다는 것이 핵심 특징이다.

---

## 동작 흐름

### 읽기 (Read)

#### 캐시 Hit 시나리오
```
클라이언트
   ↓
[Cache-Aside Logic]
   ↓
캐시 조회 → Hit!
   ↓
캐시된 데이터 반환
```

#### 캐시 Miss 시나리오
```
클라이언트
   ↓
[Cache-Aside Logic]
   ↓
캐시 조회 → Miss
   ↓
DB 조회 (비용)
   ↓
데이터 캐시 적재 (TTL 설정)
   ↓
데이터 반환
```

### 쓰기 (Write)

```
클라이언트 (Update 요청)
   ↓
DB 업데이트
   ↓
캐시 무효화 (Delete)
   ↓
다음 읽기 요청 시 자동 갱신
```

**핵심**: 쓰기는 DB가 신뢰할 수 있는 소스(Source of Truth)가 되고, 캐시는 보조 역할만 수행한다.

---

## 다른 캐싱 전략과의 비교

| 전략 | 읽기 성능 | 데이터 일관성 | 운영 복잡도 | 캐시 워밍 | 활용도 |
|------|---------|----------|---------|---------|--------|
| **Cache-Aside** | 중간 (첫 요청 느림) | 약함 (Stale 위험) | 낮음 | 필요 | 실제 사용 데이터만 |
| **Read-Through** | 높음 | 중간 | 중간 | 자동 | 데이터 저장소 관리 |
| **Write-Through** | 낮음 | 강함 | 높음 | 필요 | 쓰기 일관성 중요 |
| **Write-Behind** | 높음 | 약함 | 높음 | 필요 | 높은 쓰기 처리량 |

- **Read-Through**: 캐시 라이브러리가 miss 시 자동으로 DB 조회. 애플리케이션 코드가 간결하지만, 캐시 라이브러리가 DB 접근 방식을 알아야 한다.
- **Write-Through**: 쓰기 시 캐시와 DB를 동시에 업데이트. 일관성은 강하지만 응답 시간이 길어진다.
- **Write-Behind (Write-Back)**: 먼저 캐시에만 쓰고, 나중에 비동기로 DB 동기화. 응답 속도는 빠르지만 데이터 손실 위험이 있다.

**Cache-Aside는 가장 유연하고 구현이 간단**하면서도, **구현자가 일관성 관리를 책임져야 한다는 트레이드오프**를 가지고 있다.

---

## 장점

### 1. 유연한 데이터 선택

**실제로 자주 접근하는 데이터만 캐싱**하므로 메모리를 효율적으로 사용한다.
- 모든 데이터를 무조건 캐시하는 것이 아니라, 핫 데이터에만 선택적 적용 가능
- 캐시 저장소 비용 절감

### 2. 캐시 장애에 대한 내결함성

캐시가 다운되어도 **DB에서 직접 읽기**만 하면 되므로, 시스템이 계속 동작한다.
- 성능은 저하되지만 가용성은 유지
- Read-Through나 Write-Through는 캐시 장애 시 전체 시스템 중단 위험

### 3. 유연한 TTL(Time To Live) 관리

데이터 특성별로 다양한 만료 시간을 설정 가능하다.
- 자주 변경되는 데이터: 짧은 TTL
- 거의 변경되지 않는 데이터: 긴 TTL 또는 수동 갱신

### 4. 개발 복잡도 낮음

캐시 로직이 애플리케이션 코드에 명시적으로 드러나므로 **이해하고 디버깅하기 쉽다**.

---

## 단점 및 주의점

### 1. Cold Start (캐시 워밍 필요)

서비스 시작 직후나 캐시 리셋 후 모든 요청이 캐시 미스가 되어 **응답 시간이 크게 증가**한다.

**해결 방법**:
- **사전 로딩 (Pre-warming)**: 서비스 시작 시 자주 접근하는 데이터를 미리 캐시
- **관리 API 제공**: 운영자가 필요시 캐시를 수동으로 갱신 가능하게 구현
- **점진적 워밍**: 배포 후 트래픽을 단계적으로 증가

### 2. Stale Data (캐시 데이터 불일치)

DB는 업데이트되었는데 캐시가 아직 만료되지 않으면 **구 데이터를 제공**한다.

```
17:00 - 상품 가격 1,000원으로 업데이트
       → 캐시 무효화 완료
17:00:01 - 사용자 A가 조회 → 캐시 미스 → DB에서 1,000원 조회, 캐시 갱신
17:00:02 - 사용자 B가 조회 (TTL 1분 설정) → 캐시 Hit → 1,000원 반환
17:00:03 - DB 직접 조회 → 1,000원 확인

BUT, 만약 캐시 무효화를 놓쳤다면?
17:00 - 상품 가격 1,000원으로 업데이트 (캐시 무효화 안 함!)
17:00:01~17:00:59 - 모든 사용자가 구 가격 900원을 받음
17:01:00 - TTL 만료 → 다음 요청부터 1,000원 반환
```

**해결 방법**:
- **명시적 캐시 무효화 강제**: DB 업데이트 로직 바로 후에 캐시 삭제 코드를 **즉시 실행**하도록 설계
- **TTL 최소화**: 데이터 변경 빈도에 맞춰 적절한 TTL 설정
- **백그라운드 갱신**: TTL 종료 전 자동으로 데이터 갱신 (Refresh-Ahead)
- **이벤트 기반 무효화**: 도메인 이벤트 발행 → 별도 리스너가 캐시 삭제

### 3. Cache Stampede (캐시 폭풍)

**많은 스레드/요청이 동시에 캐시 미스를 감지**하고 모두 DB를 조회하려는 현상이다.

```
10,000명의 사용자가 동시에 같은 상품 조회
   ↓
모두 캐시 미스 감지
   ↓
10,000개의 DB 쿼리가 동시 실행
   ↓
DB 연결 풀 고갈 / CPU 스파이크 / 타임아웃
```

**해결 방법**:

#### a) **Lock 기반 접근** (Locking)
```
첫 번째 요청: 캐시 미스 감지 → Lock 획득 → DB 조회 → 캐시 저장 → Lock 해제
다른 요청들: 캐시 미스 감지 → Lock 대기 → Lock 해제 후 캐시에서 읽기
```
- 안전하지만, Lock 대기 시간에 따라 응답 지연 가능

#### b) **Probabilistic Early Expiration** (확률적 조기 갱신)
```
TTL을 두 부분으로 나눔
- TTL = 100초
- Soft TTL = 80초 (80% 시점)

80초 경과 후 일정 확률(예: 5%)로 백그라운드 갱신 시작
-> 대부분의 요청은 캐시 히트, 극소수만 DB 조회
-> 캐시 폭풍 가능성 대폭 감소
```

#### c) **Xfetch Algorithm** (계단식 갱신)
TTL 만료 시점을 랜덤하게 분산
```
100명이 동시에 접근해도 갱신 시점이 다르면
분산된 DB 부하로 스파이크 완화
```

## 언제 쓰면 좋은가?

### 추천 시나리오

1. **변경이 적은 데이터**
   - 상품 카테고리, 설정값, 마스터 데이터
   - 올리브영 사례: 상품 정보, 카테고리, 이미지 URL 등

2. **읽기 작업이 대부분**
   - 쓰기는 드물지만 읽기가 매우 빈번한 경우
   - 예: 통계, 레포팅, 추천 데이터

3. **응답 시간이 중요**
   - 밀리초 단위의 응답이 필요한 서비스
   - 반복적 DB 조회로 인한 오버헤드 제거 필요

4. **캐시 장애 허용 가능**
   - 캐시 다운 시 성능 저하는 있지만, 전체 시스템이 멈추면 안 되는 경우

### 부적합 시나리오

- 실시간 일관성이 절대적으로 필요한 경우 (금융 거래, 재고 관리)
- 모든 데이터를 캐싱해야 하는 경우 (메모리 비효율)
- 쓰기 작업이 매우 빈번한 경우

## 실제 사례: 올리브영 MSA 환경

올리브영 기술 블로그의 "MSA 환경에서 도메인 데이터 연동 전략" 글에서 소개한 사례를 보면:

**변경이 적은 데이터에 Cache-Aside 패턴 적용**

```
상품 정보, 카테고리, 옵션 데이터 (변경 빈도: 하루 1~2회)
   ↓
Redis Cache-Aside + TTL 1시간 설정
   ↓
첫 접근 시 캐시 미스 → DB 조회 (비용 발생)
다음 59분간 → 캐시 히트 (매우 빠름)
   ↓
변경 발생 시 명시적으로 캐시 키 삭제
다음 요청 시 자동으로 최신 데이터 캐싱
```

이 전략으로 **불필요한 API 호출을 줄이면서도 최신 데이터를 보장**하고, 실시간성이 필요한 데이터는 별도로 Kafka 이벤트 기반으로 처리했다.
