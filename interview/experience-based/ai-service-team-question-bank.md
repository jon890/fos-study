## 1분 자기소개 준비

- NHN에서 4년째 Spring Boot 기반 백엔드 개발을 해왔고, 최근 1년 반은 AI 서비스팀에서 RAG 파이프라인 설계와 AI 툴링 도입을 주도했습니다.
- Confluence 문서를 OpenSearch에 벡터 색인하는 배치 파이프라인을 처음부터 설계했는데, 11개 Step 실패 격리와 AsyncItemProcessor를 활용한 I/O 병렬화, 커서 기반 재시작 가능 설계가 핵심이었습니다.
- 그 전에는 소셜 카지노 게임팀에서 StampedLock과 RabbitMQ Fanout Exchange로 다중 서버 캐시 정합성을 해결하고, Transactional Outbox Pattern으로 Kafka 메시지 유실을 방지하는 설계를 직접 구현했습니다.
- 단순 기능 구현을 넘어 팀 전체의 개발 속도를 높이는 구조와 도구를 만드는 것을 중요하게 여깁니다. Cursor Rules 체계를 구축해 신규 게임 3종을 에이전트 단독으로 구현하고 팀 전체에 전파해 반복 개발 사이클을 단축했습니다.

## 올리브영/포지션 맞춤 연결 포인트

- 제가 구현한 OpenSearch 증분 색인·삭제 동기화 파이프라인과 올리브영 기술 블로그의 Cache-Aside + Kafka 하이브리드 설계는 대규모 데이터를 실시간으로 처리하는 동일한 문제를 서로 다른 도메인에서 풀고 있어, 실무 감각이 곧바로 연결됩니다.
- Kafka 기반 Transactional Outbox Pattern 설계 경험이 올리브영 Event-Driven Architecture 도메인 간 연동 구조에 직접 적용될 수 있습니다.
- 다중 서버 환경에서 StampedLock과 RabbitMQ Fanout Exchange로 캐시 정합성을 직접 해결한 경험이 1,600만 고객 규모 트래픽에서의 캐싱 전략 설계 토론에 실질적 기여가 됩니다.
- Spring Batch 기반 대용량 처리와 OpenSearch 벡터 색인 운영 경험이 상품 검색 파이프라인 도메인에 곧바로 적용 가능합니다. 목적은 다르지만 증분 처리·삭제 동기화 패턴은 동일합니다.

## 지원 동기 / 회사 핏

### 왜 이직하려는가
- AI 서비스팀 파이프라인 설계를 통해 대용량 처리 설계·구현 역량은 쌓았지만, 1,600만 고객 규모의 실시간 커머스 트래픽에서 이 역량이 어떻게 동작하는지 직접 경험하고 싶었습니다.
- 게임 도메인은 세션 기반 트래픽 패턴이 비교적 예측 가능하지만, 커머스는 올영세일처럼 예측 불가한 피크 트래픽과 복잡한 비즈니스 규칙이 공존합니다. 이 환경에서 설계 근육을 키우고 싶어 이직을 결심했습니다.
- 현재 조직에서 AI 툴링 도입·파이프라인 설계 등 새로운 영역을 개척하는 역할을 해왔는데, 더 큰 도메인 복잡성과 사용자 규모가 있는 환경에서 성장하고 싶었습니다.

### 왜 올리브영인가
- 올리브영 기술 블로그를 통해 Feature Flag, Shadow Mode, Resilience4j Circuit Breaker 같은 패턴을 실제 올영세일 트래픽에서 운영 중이라는 것을 확인했고, 이런 수준의 기술적 고민을 함께 하는 조직에서 일하고 싶었습니다.
- 올영세일 중 무중단 OAuth2 전환을 P95 50ms, 성공률 100%로 달성한 사례는 기술 역량뿐 아니라 리스크를 관리하며 배포하는 조직 문화를 보여줍니다. 그 문화 안에서 함께 일하고 싶습니다.
- 1,600만 고객에게 직접 영향을 주는 서비스에서 캐싱·메시징·대용량 처리 역량을 발휘하고 싶습니다. 제가 설계한 OpenSearch 파이프라인 경험이 상품 검색 도메인에 바로 기여할 수 있다고 판단했습니다.

### 왜 이 역할에 맞는가
- 커머스플랫폼 백엔드 개발 포지션은 MSA 환경에서의 도메인 간 데이터 연동, 캐시 전략, 이벤트 드리븐 설계를 요구하는데, 이 세 가지가 제가 실무에서 가장 많이 고민하고 직접 구현한 영역입니다.
- Spring Batch, Kafka, OpenSearch, Redis를 조합해 실제 운영 파이프라인을 설계한 경험이 상품 검색·전시 로직 개발에 즉각 기여할 수 있습니다.
- 팀 전체가 빠르게 기여할 수 있는 구조를 만드는 것을 중요하게 여기는데, 웰니스개발팀이 이 가치를 공유하는 팀이라고 기술 블로그를 통해 느꼈습니다.

## 메인 질문 1. Confluence 색인 배치를 11개 Step으로 나눈 설계 의도를 설명해 주세요. 왜 하나의 Step 안에서 모든 처리를 하지 않았나요?

> 추가: 2026-04-18 | 업데이트: 2026-04-18

### 면접관이 실제로 보는 것

- Step 분리의 실질적 운영 이점(실패 격리, 재시작 가능성)을 트레이드오프와 함께 설명할 수 있는지 확인 — '좋은 구조'가 아닌 실제 장애 시나리오를 근거로 답하는지 본다
- 설계 결정을 구체적 사례(임베딩 API 장애 시 동작, 청크 커밋 단위 이력 저장)로 뒷받침할 수 있는지 확인

### 실제 경험 기반 답변 포인트

- 실패 격리: 댓글 색인 Step이 임베딩 API 장애로 실패해도 페이지 색인 Step 결과는 DB에 보존되어 재시작 시 댓글 Step부터만 재실행 가능
- 재시작 가능성: Spring Batch가 BATCH_JOB_EXECUTION 테이블에 Step 단위 실행 이력을 저장하므로 JVM 재시작·서버 장애 상황에서도 실패 지점을 정확히 알 수 있음 — try-catch로는 얻을 수 없는 영속적 재시작 지점
- 단일 책임 분리: 수집·변환·색인·삭제 동기화를 각 Step으로 나눠 코드 수정 범위를 최소화하고 각 Step을 독립적으로 테스트 가능
- 트레이드오프 인지: Step이 늘어날수록 Step 간 컨텍스트 공유 설계가 복잡해지고 @JobScope 빈 재시작 시 allowStartIfComplete 같은 운영 고려사항이 추가됨

### 1분 답변 구조

- 핵심 이유는 실패 격리와 재시작입니다. 임베딩 API 장애 시 페이지 색인 Step만 실패하고 이미 완료된 삭제 동기화 결과는 살아있어, 복구 후 실패한 Step부터만 이어서 실행할 수 있었습니다.
- Spring Batch는 Step 단위로 실행 이력을 DB에 저장합니다. 하나의 큰 Step 안에서 try-catch로 처리하면 JVM 재시작 후 처음부터 재처리해야 하고 임베딩 API 비용이 중복 발생합니다.
- 트레이드오프로 Step이 늘어날수록 컨텍스트 공유 설계 비용이 커집니다. @JobScope 홀더 빈을 두고 상태 로더 Step에 allowStartIfComplete를 적용해 이 문제를 해결했습니다.

### 압박 질문 방어 포인트

- '그냥 try-catch로 실패 처리하면 되지 않나요?'라는 질문에는 — try-catch는 코드 레벨 예외 처리이고 Step 분리는 JVM 재시작·서버 장애 상황에서도 DB 기반으로 어디까지 처리됐는지를 보장합니다. 재시작 범위와 비용이 다른 문제입니다.
- '11개는 과설계 아닌가요?'라는 질문에는 — 각 Step의 실패 격리 단위가 운영에서 실제로 가치를 발휘했습니다. 삭제 동기화 Step 실패 시 색인 결과를 버리지 않아도 됐고, 재시작 단위를 거칠게 합치면 임베딩 API 비용이 중복 발생합니다.

### 피해야 할 약한 답변

- 'Clean Code 원칙에 따라 단일 책임으로 나눴습니다'라고만 답하면 — 운영 관점의 실패 격리와 재시작 이점을 이해하지 못한 것으로 보임
- 'Spring Batch 튜토리얼 예제가 그렇게 되어 있어서요'라고 답하면 — 설계 판단 근거가 전혀 없는 것으로 보임

**F1-1.** Step 간 데이터 공유를 JobExecutionContext 대신 @JobScope 빈으로 전환한 이유가 무엇인가요?

**F1-2.** allowStartIfComplete(true)를 설정해야 했던 Step은 어떤 Step들이고, 설정하지 않으면 구체적으로 어떤 문제가 생기나요?

**F1-3.** 11개 Step 중 설계가 가장 어려웠던 Step은 무엇이고 왜 어려웠나요?

**F1-4.** 배치 잡 전체 실행 시간은 어느 정도였고 성능 병목은 어디에 있었나요?

## 메인 질문 2. 임베딩 API 호출을 AsyncItemProcessor로 병렬화한 설계 결정을 설명해 주세요. 동기 방식 대비 어떤 이점이 있었고, 병렬 처리에서 주의해야 할 점은 무엇인가요?

> 추가: 2026-04-18 | 업데이트: 2026-04-18

### 면접관이 실제로 보는 것

- I/O 바운드와 CPU 바운드 작업의 차이를 이해하고 적절한 병렬화 전략을 선택한 근거를 설명할 수 있는지 확인
- AsyncItemProcessor + AsyncItemWriter 조합의 내부 동작(Future 기반, Writer 단에서의 취합)과 스레드 안전성 설계를 실제로 이해하는지 확인

### 실제 경험 기반 답변 포인트

- I/O 바운드 특성: 임베딩 API 호출은 네트워크 대기 시간이 대부분이므로 동기 처리 시 청크 하나 처리에 청크 사이즈 × 응답 시간(약 200ms)이 소요되어 수천 페이지 처리 시 수 분이 걸릴 수 있었음
- AsyncItemProcessor 동작 원리: 각 아이템을 스레드풀에 제출해 Future로 반환, AsyncItemWriter가 Future.get()으로 결과를 수집한 뒤 벌크 색인 — Processor 단계에서 I/O 대기 시간이 겹쳐 실행됨
- 스레드 안전성 확보: @StepScope로 Processor 인스턴스를 Step 실행당 하나로 한정하고, 내부에 공유 변경 가능 상태를 두지 않아 별도 동기화 없이 안전
- 스레드풀 크기 결정: 임베딩 API Rate Limit을 초과하지 않는 범위로 제한 — 너무 크면 Rate Limit 초과, 너무 작으면 병렬화 이득 감소

### 1분 답변 구조

- 임베딩 API는 네트워크 I/O가 전부라 동기 처리 시 청크 하나 처리에 수십 초가 걸릴 수 있었습니다. AsyncItemProcessor로 청크 내 아이템을 스레드풀에서 동시에 처리해 I/O 대기 시간을 겹쳐 실행했습니다.
- 내부적으로 AsyncItemProcessor가 Future를 반환하고 AsyncItemWriter가 모든 Future를 get()으로 기다린 뒤 결과를 벌크 색인합니다. Writer 단에서 기다리는 것이지, Processor 단에서는 병렬로 실행되어 I/O 대기 시간이 겹쳐집니다.
- 스레드 안전성은 Processor 내부에 공유 상태를 두지 않는 방식으로 확보했고, 스레드풀 크기는 임베딩 API Rate Limit을 초과하지 않는 범위에서 결정했습니다.

### 압박 질문 방어 포인트

- 'Processor에서 스레드 안전성을 어떻게 보장했나요?'라는 질문에는 — Processor를 @StepScope로 Step 실행당 하나씩 생성하고 내부에 공유 변경 가능 필드를 두지 않았습니다. 외부 API 호출만 있고 인스턴스 필드 변경이 없으니 별도 동기화가 필요 없었습니다.
- 'CompletableFuture로 직접 구현하면 안 되나요?'라는 질문에는 — 가능하지만 AsyncItemProcessor + AsyncItemWriter는 Spring Batch의 청크 트랜잭션과 재시작 메커니즘에 통합되어 있어, 직접 구현 시 이 통합을 별도로 구현해야 하고 실패 재처리 로직이 복잡해집니다.

### 피해야 할 약한 답변

- '빠르게 하려고 멀티스레딩을 썼습니다'라고만 답하면 — I/O 바운드 특성 분석이나 스레드 안전성 고려 없이 기계적으로 적용한 것처럼 보임
- 'Future.get()으로 기다리니까 동기랑 다를 게 없지 않나요?'라는 반문에 '맞는 것 같습니다'라고 후퇴하면 — Writer에서 기다리는 것이고 Processor 단에서 병렬 실행되어 I/O 대기가 겹쳐짐을 설명 못 한 것으로 보임

## 메인 질문 3. Spring Batch 파이프라인에서 배치 중간 실패 시 마지막 처리 지점부터 재시작 가능하도록 설계했다고 하셨는데, 구체적으로 어떻게 구현했나요? @JobScope 빈을 쓸 때 재시작 시 NPE가 생겼다고 하셨는데 근본 원인과 해결책을 설명해 주세요.

> 추가: 2026-04-18 | 업데이트: 2026-04-18

### 면접관이 실제로 보는 것

- Spring Batch의 재시작 메커니즘(ItemStream, ExecutionContext)을 개념이 아닌 실제 구현 레벨로 이해하는지 확인
- @JobScope의 내부 동작(CGLIB 프록시, Job 생명주기 범위)과 재시작 시 주의사항을 경험 기반으로 설명할 수 있는지 확인

### 실제 경험 기반 답변 포인트

- ItemStream 구현: Reader에 ItemStream을 구현해 현재 페이지 커서 위치를 update()에서 ExecutionContext에 저장, 재시작 시 open()에서 저장된 위치를 읽어 페이지네이션 재개 — 이미 처리한 페이지 건너뜀
- @JobScope 내부 동작: 싱글톤 빈에 CGLIB 프록시가 주입되고 실제 호출 시 현재 Job 스코프 인스턴스로 위임, 재시작 시 새 JobExecution으로 새 인스턴스가 초기화되어 상태가 비어있는 상태로 시작
- allowStartIfComplete(true) 설정 이유: 상태를 채우는 Step(스페이스 수집, 페이지 ID 수집)이 COMPLETED 처리된 채로 스킵되면 @JobScope 빈이 빈 상태로 남아 NPE 발생 — 이 Step들만 재시작 시에도 강제 재실행
- JobExecutionContext vs @JobScope 홀더 분리: 수천 개 페이지 ID를 ExecutionContext에 저장하면 청크 커밋마다 DB 직렬화 부하 발생 — ExecutionContext는 커서 같은 경량 상태 전용, 대량 데이터는 @JobScope 홀더 빈으로 분리

### 1분 답변 구조

- Reader에 ItemStream을 구현해 페이지네이션 커서 위치를 ExecutionContext에 저장했습니다. 실패 후 재시작하면 저장된 커서 위치부터 API 호출을 재개해 이미 처리한 페이지를 건너뜁니다.
- @JobScope 빈은 Job마다 새 인스턴스가 생성되므로 재시작 시 상태 로더 Step이 COMPLETED 상태라도 반드시 재실행되어야 합니다. allowStartIfComplete(true)로 이를 보장했습니다. 이 설정 없이 재시작하면 빈이 빈 상태로 남아 NPE가 발생했습니다.
- 처음에 수천 개 페이지 ID를 JobExecutionContext에 저장했다가 청크 커밋마다 DB 직렬화 부하가 생겨 @JobScope 홀더 빈으로 옮겼습니다. ExecutionContext는 커서 같은 경량 상태 전용으로 용도를 제한했습니다.

### 압박 질문 방어 포인트

- 'JobExecutionContext와 @JobScope 빈의 차이가 무엇인가요?'라는 질문에는 — ExecutionContext는 청크 커밋마다 DB에 직렬화되어 JVM 재시작 후에도 영속, @JobScope 빈은 인메모리에서 JVM 내에서만 유효합니다. 재시작 커서처럼 영속이 필요한 경량 상태는 ExecutionContext, 대량 데이터는 @JobScope 빈이 적합합니다.
- 'allowStartIfComplete를 모든 Step에 걸면 되지 않나요?'라는 질문에는 — 이미 완료된 색인·삭제 Step까지 재실행되면 불필요한 임베딩 API 비용이 발생합니다. 상태 로더 Step만 선택적으로 설정해야 합니다.

### 피해야 할 약한 답변

- '재시작 기능은 Spring Batch가 자동으로 해줍니다'라고만 답하면 — ItemStream 구현, allowStartIfComplete 설정 등 개발자가 직접 해야 하는 부분을 이해하지 못한 것으로 보임
- 'JobExecutionContext에 모든 상태를 저장하면 됩니다'라고 답하면 — 대량 데이터를 청크 커밋마다 직렬화하는 DB 부하 문제를 전혀 모르는 것으로 보임

## 메인 질문 4. OCR 서버 배포·스케일인 시 503 에러를 분석하고 수정한 과정을 설명해 주세요. 처음에 어떤 단서로 Graceful Shutdown 문제라고 판단했나요?

> 추가: 2026-04-18 | 업데이트: 2026-04-18

### 면접관이 실제로 보는 것

- 장애 분석 시 에러 코드·응답 헤더·발생 패턴에서 가설을 세우고 검증하는 체계적 사고를 볼 수 있는지 확인
- 컨테이너 종료 시퀀스(preStop → SIGTERM → 프로세스 종료)와 Graceful Shutdown의 관계를 운영 경험 기반으로 설명할 수 있는지 확인

### 실제 경험 기반 답변 포인트

- 단서 분석 과정: error 111은 ECONNREFUSED(TCP 레벨 연결 거부), 응답 헤더 server: envoy는 Envoy 자체는 살아있고 upstream(:50051) 연결만 실패라는 의미, 에러가 30~60초 주기로 묶음 발생 후 자연 소멸 → 배포·스케일인 이벤트와 정확 일치로 컨테이너 종료 시퀀스 문제로 확정
- 근본 원인 두 가지: Python gRPC 서버에 SIGTERM 핸들러가 없어 SIGTERM 수신 즉시 종료(server.wait_for_termination()만 있고 signal.signal 없음), preStop sleep 20s 동안 Envoy가 살아있어 죽은 50051로 계속 연결 시도
- NCS 제약: terminationGracePeriodSeconds 30초 고정, API 스펙에 해당 필드 없어 변경 불가 — 30초 예산 내에서 preStop 15s + gRPC grace 12s + 여유 3s로 시간 배분
- 3지점 수정: SIGTERM 핸들러 추가(server.stop(grace=12)), supervisord stopwaitsecs=17 설정(grace 12s + 여유 5s), preStop sleep 20s → 15s 단축

### 1분 답변 구조

- 에러 로그에서 두 가지 단서가 있었습니다. error 111은 ECONNREFUSED이고 응답 헤더에 server: envoy가 있었습니다. Envoy는 살아있는데 upstream gRPC 서버(:50051)에 연결이 거부된 것입니다.
- 에러가 30~60초 주기로 묶음 발생하고 자연히 사라지는 패턴이 배포·스케일인 이벤트와 정확히 일치해 컨테이너 종료 시퀀스 문제로 판단했습니다. preStop sleep 동안 Envoy가 이미 죽은 gRPC 서버에 연결을 시도하는 상황이었습니다.
- 수정은 SIGTERM 핸들러 추가(server.stop(grace=12))와 supervisord stopwaitsecs=17 설정, preStop sleep을 15초로 단축해 NCS의 30초 고정 종료 예산 안에서 해결했습니다.

### 압박 질문 방어 포인트

- 'preStop을 없애고 Kubernetes가 직접 SIGTERM을 보내게 하면 안 되나요?'라는 질문에는 — preStop은 Envoy의 drain_listeners를 호출해 신규 연결을 먼저 차단하는 역할입니다. 이 없이 SIGTERM을 먼저 받으면 Envoy가 drain 없이 새 요청을 받아 in-flight 요청이 유실될 수 있습니다.

### 피해야 할 약한 답변

- '503 에러가 나서 서버를 재배포했습니다'라고만 답하면 — 원인 분석 없이 증상만 해소한 것으로 보임, 동일 문제가 반복될 것

- Spring Batch Step 분리의 실패 격리·재시작 이점을 임베딩 API 장애 시나리오 같은 구체적 사례와 함께 1분 이내로 설명할 수 있는가
