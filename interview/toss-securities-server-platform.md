## 1. 채용 공고 상세

### 팀 소개

Server Platform 팀은 토스증권의 여러 사일로 조직이 공통으로 필요로 하는 **공통 기능과 프레임워크를 개발**하여 지원하는 팀이다. 올해의 목표는 증권 서비스들이 더 빠르게 출시되고 안정적으로 운영될 수 있도록 기반을 닦는 것이다.

일반 사일로 팀과 다르게 **전담 PM이 없으며**, 엔지니어가 직접 전략과 기능 우선순위를 결정한다.

## 6. 면접 준비 포인트

### 기술 면접 예상 주제

- [ ] JVM 내부 동작 (GC 알고리즘, Heap 구조, ClassLoader)
- [ ] Spring Framework 내부 동작 (BeanFactory, AOP, 트랜잭션 동작 원리)
- [ ] 공통 라이브러리/프레임워크 설계 원칙 (OCP, DIP, 인터페이스 추상화)
- [ ] 성능 프로파일링 방법론 (CPU 프로파일링, 메모리 덤프 분석)
- [ ] 분산 시스템 트러블슈팅 (분산 락, 일관성 모델)
- [ ] Kafka 운영 경험 (파티셔닝, 컨슈머 그룹, 재처리 전략)
- [ ] Redis 운영 경험 (클러스터 구성, 메모리 관리, 캐시 무효화 전략)
- [ ] 공통 테스트 인프라 설계 경험

### 내 경험 매핑

| 공고 요구사항 | 내 경험 |
| --- | --- |
| 공통 기능·프레임워크 개발 | SlotTemplate 추상화 — 5개 게임 공통 엔진 통합, AbstractSlotTest 테스트 공통 템플릿 |
| 성능 프로파일링·최적화 | Alias Method O(1) 가중 랜덤 교체, Welford 알고리즘으로 OOM 해결, 시뮬레이터 ThreadLocal 레이스컨디션 수정 |
| 다중 레이어 트러블슈팅 | Spring, JVM(ThreadLocal), 동시성(AtomicReference), 메모리 누수(List→스트리밍) 다층 디버깅 경험 |
| 라이브러리·관행 표준화 | Cursor Rules 20+ 작성으로 AI 에이전트용 아키텍처 표준 문서화, ReactiveSimulator 공통 시뮬레이터 마이그레이션 |
| Redis 운영 경험 | Ehcache + Redis(Streams/JSON/Search) 다양한 캐시 패턴 설계, MQ Fanout 캐시 동기화 |
| Spring Framework 숙련 | Spring Boot 2.6 → 3.x, Spring Batch 11단계 파이프라인, JPA 실무 다수 |
| PM 없이 자체 우선순위 결정 | 슬롯 엔진 추상화 프로젝트 자발적 설계·추진 경험 |

### 강점 분석 (Platform 적합도 높은 이유)

1. **공통 추상화 경험**: 5개 이상의 게임에 SlotTemplate, PayCalculator 계층 설계 → 공통 프레임워크 개발 직결
2. **성능 최적화 트랙레코드**: O(1) 알고리즘 교체, OOM 해결, 레이스컨디션 수정 → 멀티 레이어 성능 프로파일링 역량
3. **테스트 인프라 설계**: AbstractSlotTest + JUnit5 Extension으로 팀 전체 테스트 생산성 향상 → 공통 인프라 기여
4. **문서화·표준화 DNA**: Cursor Rules 20+ 작성, 공통 시뮬레이터 마이그레이션 → 조직 내 표준 수립 경험
5. **자율적 기술 의사결정**: PM 없이 슬롯 엔진 리팩터링 설계·실행

### 상대적 약점 (보완 필요)

- JVM 심층 튜닝(GC 알고리즘 교체, 힙 덤프 분석) 직접 경험 부족
- Kafka 운영(파티셔닝 전략, 컨슈머 그룹 재처리) — Redis Streams로 유사 경험 강조 가능
- 사내 공통 라이브러리 배포·버전 관리 경험 없음

## 7. Product vs Platform 비교 분석

| 항목 | Product | Platform |
| --- | --- | --- |
| 주 업무 | 유저 대면 서비스 기능 개발 | 공통 프레임워크·라이브러리 개발 |
| 도메인 | 증권(주식, 계좌, 시세, 커뮤니티) | 개발자 플랫폼(인프라, 공통 도구) |
| PM 유무 | 있음 | 없음 (엔지니어 직접 결정) |
| 내 경험 적합도 | ★★★☆ (기술 스택 일치, 도메인 갭 존재) | ★★★★ (공통화/추상화/성능 최적화 직결) |
| 성장 방향 | 증권 도메인 깊이 + 고트래픽 서비스 | 플랫폼 엔지니어링 + 조직 레버리지 |

**권고**: Platform 직무가 현재 경험(SlotTemplate 추상화, 성능 최적화, 테스트 인프라, 표준화)과 더 직접적으로 매핑됨. Product는 증권 도메인 적응 기간이 필요하나 기술 스택은 동일하게 경쟁력 있음.
