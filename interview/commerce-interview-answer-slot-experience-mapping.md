# [초안] 커머스/F&B 면접 답변집 — 슬롯 도메인 경험을 주문·결제·쿠폰·매장 설계로 매핑하기

## 왜 이 문서가 필요한가

CJ푸드빌 디지털 채널 백엔드 면접에 슬롯·베팅·AI 배치 같은 이력으로 들어가면, 면접관이 처음 30초 안에 결정하는 한 가지가 있다. "이 사람이 자기 경험을 *우리 도메인 언어*로 다시 말할 수 있는 사람인가, 아니면 회사 단어만 모르는 게 아니라 사고의 축 자체가 어긋난 사람인가." 슬롯 RTP·RCC·페이라인 같은 단어를 그대로 던지면 그림이 안 그려진다. 반대로 "비슷할 거예요"로 도망가면 도메인 깊이 점수를 못 받는다.

이 문서는 그 사이를 메우는 *면접 답변 playbook*이다. 매핑의 *개념과 6축*은 [CJ푸드빌 디지털 채널 면접](../architecture/cj-foodville-commerce-domain-design-interview.md)에서 정리했으므로 여기서는 복습하지 않고, *실제 면접에서 1분 안에 말할 답변 + 꼬리 질문 방어*에 집중한다. 자기소개 60초, "왜 우리 회사인가" 60초, "커머스 도메인 처음 아니냐" 압박, 그리고 SlotTemplate / RCC / StampedLock / AliasMethod / Kafka Outbox / 캐시 정합성을 *주문·결제·쿠폰·메뉴·매장 정책* 설계로 옮겨 답하는 6세트가 본 문서의 본체다.

## 매핑 한 줄 요약: 도메인 단어는 바뀌어도, 의사결정 축은 같다

면접에서 받게 될 모든 슬롯 → 커머스 답변은 다음 6축 중 하나 이상으로 정리된다. 답변 도중 길을 잃으면 항상 이 축으로 돌아온다.

- 어디까지를 하나의 Aggregate로 묶고, 어디부터 이벤트로 끊을 것인가
- 어떤 상태 전이가 불법인지 enum + 전이 규칙으로 강제할 수 있는가
- 동시 요청이 같은 자원에 부딪힐 때 락 / 유니크 키 / 낙관적 락 / 분산 락 중 무엇을 고를 것인가
- 트랜잭션 커밋 이후 외부 시스템 전파를 어떤 패턴으로 보장할 것인가
- 캐시와 진실 원천이 어긋날 때 어떻게 좁히고 어떻게 관측할 것인가
- 장애가 났을 때 1차 대응에서 무엇을 보고, 무엇을 끄고, 무엇을 재시도할 것인가

이 6축은 슬롯 도메인에서 이미 7년차 수준으로 다뤄왔고, 커머스 도메인의 단어로 옮길 때도 같은 모양 그대로 들어간다.

## 1분 자기소개 (커머스/F&B 톤으로 다시 짠 버전)

> 약 7년차 Java 백엔드 개발자 김병태입니다. 가장 가까운 경험은 NHN NSC 슬롯팀에서 운영형 자사 서비스 백엔드를 만든 일입니다. 슬롯 5종 이상이 누적된 시점에 공통 흐름을 `SlotTemplate` / `BaseSlotService`로 추상화했고, 단기 RTP 편차를 보정하는 RCC라는 백그라운드 캐시 시스템을 직접 설계했습니다. 다중 서버 인메모리 캐시 정합성 문제는 Hibernate `PostCommitUpdateEventListener` → RabbitMQ Fanout → `StampedLock`까지 단일 스택으로 풀었고, 비동기 후처리는 `@TransactionalEventListener(AFTER_COMMIT)` + `REQUIRES_NEW`로 메시지 유실 없는 Outbox 구조로 운영했습니다. 그 뒤 NHN AI 서비스팀에서 Confluence → OpenSearch 벡터 색인 배치 파이프라인을 11 Step / `AsyncItemProcessor` 기반으로 처음부터 설계·구현했고, 운영 중인 OCR gRPC 서버의 503 graceful shutdown 문제를 30초 종료 예산 안에서 preStop·SIGTERM 핸들러·supervisord 조합으로 해결했습니다. CJ푸드빌 디지털 채널이 다루는 주문·결제·쿠폰·매장 운영 도메인의 핵심 어려움이 *상태 전이 안전성 / 캐시 정합성 / 트랜잭션-이벤트 경계 / 운영 관측성*이라고 이해하고 있고, 그 네 가지는 제가 직접 풀어본 문제와 1:1로 겹칩니다.

## "왜 CJ푸드빌인가" 60초 답변

> 두 가지 이유입니다. 첫째, *운영형 자사 서비스 백엔드*라는 자리입니다. 저는 외주형이 아니라 자기 서비스를 길게 키우는 환경에서 가장 잘 일해왔습니다. 슬롯팀에서 RCC를 두 분기에 걸쳐 5종 슬롯에 확장하고, 캐시 정합성 문제를 사후 대응이 아니라 구조로 해결하면서 운영을 안정화한 경험이 그렇습니다. 둘째, *레거시와 신규의 공존*입니다. 디지털 채널은 JSP/jQuery 기반 화면과 모바일 앱 API, SPA가 한 백엔드를 공유합니다. 저는 슬롯팀에서 어드민 / 게임 클라이언트 / 사내 시뮬레이터가 같은 도메인 서비스를 다른 진입점으로 호출하는 구조를 다뤘고, AI 서비스팀에서 사내 RAG와 외부 모델 SDK를 동시에 붙이는 환경에서 *경계*를 어디에 그어야 하는지 판단해 봤습니다. CJ푸드빌의 점진적 마이그레이션 환경에서 그 감각이 빠르게 쓰일 것이라 생각합니다.

## 도메인 전환 우려 방어 — "커머스 처음 아니냐"

면접에서 가장 자주 나오는 압박. 정직-방어 균형이 핵심.

답변 골격(60초):

> 솔직히 *주문·결제·쿠폰* 도메인을 운영해 본 적은 없습니다. 다만 제가 슬롯 도메인에서 풀어본 문제들이 *형태 그대로* 커머스에 옮겨집니다. 예를 들어 슬롯 한 판을 Aggregate Root로 보고 통계·미션·알림을 이벤트로 끊은 결정은 주문 한 건을 Aggregate Root로 보고 적립·알림·CRM을 이벤트로 끊는 결정과 같습니다. 슬롯의 캐시 충족 판정 / 캐시 히트 / 비동기 캐시 채움 상태머신은 주문의 *결제 대기 / 결제 승인 / 매장 접수 / 픽업 대기* 상태머신과 같은 모양이고요. 단어를 바꾸는 데 1\~2주, 깊이까지 내 것으로 만드는 데 한 분기를 잡고 들어가겠습니다. 그동안은 *이미 풀어본 의사결정 축*을 가지고 팀이 이미 내린 결정을 빠르게 따라잡는 형태로 기여하겠습니다.

피해야 할 약한 답변:

- "비슷할 거예요" — 도메인 깊이 점수를 통째로 깎인다
- "공부하겠습니다"만 반복 — 4년차+ 기준 아래로 떨어진다
- 슬롯/베팅을 *깎으면서* 답변 — 본인 자산을 스스로 버리는 모양이 된다

## 기본기 검증 질문 6세트

각 세트는 *면접관이 던질 만한 질문 / 1분 답변 / 꼬리 질문 대응 / 피해야 할 약한 답변* 4단으로 구성한다.

### SlotTemplate / BaseSlotService → 주문 도메인 추상화

**질문**: "슬롯에서 했던 추상화가 주문 도메인에 어떻게 옮겨질까요?"

**1분 답변**:

> 슬롯팀에서 처음에는 슬롯마다 스핀 로직을 따로 짰는데, 5종이 누적되면서 라인·웨이·클러스터 페이 방식이 공통 축이라는 게 보였습니다. 그 시점에 `SlotTemplate`으로 페이 방식을 추상화하고 `BaseSlotService` 추상 클래스로 빈 주입이 필요한 공통 로직을 묶었습니다. 같은 사고를 커머스로 옮기면, 주문이라는 같은 흐름에 매장 픽업·배달·예약·단체주문이 들어올 때 모든 채널이 한 클래스에 if로 쌓이지 않게 `OrderFulfillmentStrategy` 같은 인터페이스로 끊고, 공통 흐름은 상위 추상 서비스에 둡니다. 결정 기준은 동일합니다 — 반복을 충분히 본 뒤에 추상화한다, default 메서드로 상태가 필요해질 때는 인터페이스 대신 추상 클래스로 내려간다.

**꼬리 질문 — "왜 처음부터 추상화하지 않나요?"**

> 처음부터 추상화하면 잘못된 경계를 그을 위험이 큽니다. 실제로 슬롯도 1\~2종일 때는 페이 방식 추상화 축이 안 보였습니다. 5종이 쌓이고 *어떤 변동점이 슬롯 고유 / 어떤 변동점이 공통*인지가 분명해지고 나서 추상화하는 편이 *지우고 다시 그리는 비용*보다 쌉니다.

**약한 답변**: "공통 로직은 무조건 추상화해야 한다" — 결정 근거 없는 일반론.

### RCC 백그라운드 캐시 → 메뉴/프로모션 캐시 사전 적재

**질문**: "RCC를 그대로 커머스에 옮길 수 있나요? 어떤 시나리오에 쓰겠습니까?"

**1분 답변**:

> RCC는 단기 RTP 편차를 보정하기 위해 *좋은 결과*를 백그라운드로 사전 계산해 DB에 적재해 두고, 필요한 순간에 캐시 히트로 반환하는 시스템이었습니다. 핵심 결정은 세 가지였습니다 — 슬롯마다 캐시 조건이 다르니 `RccSpinResultAnalyzer` 인터페이스로 슬롯별 구현체 주입, 잭팟 포함 여부는 `NoOpJackpotService`로 무동작 구현체로 다루기, 캐시 충족 판정 쿼리는 복합 인덱스로 최적화. 같은 패턴을 커머스로 옮기면, 피크 시간 *오늘의 메뉴* / *매장 가용성* / *프로모션 적용가*를 미리 계산해 캐시 테이블에 적재하고 첫 호출에서 캐시 히트로 응답한 뒤 비동기로 다음 슬롯을 채우는 형태가 됩니다. 매장별·브랜드별 정책은 `MenuAvailabilityAnalyzer` 같은 슬롯별 인터페이스로 분기 제거, 픽업 미지원 매장은 `NoOpPickupService`로 무동작 구현체.

**꼬리 질문 — "캐시 미스가 났을 때는?"**

> 일반 경로로 즉시 계산해 응답하고, 같은 키에 대한 비동기 *백그라운드 갱신*을 트리거합니다. 슬롯에서는 DB 유니크 키로 동시 캐시 생성 충돌을 막았고, 커머스에서도 매장ID + 메뉴ID + 시간버킷 단위로 같은 유니크 키 전략을 씁니다.

**약한 답변**: "Redis로 캐싱하면 됩니다" — 정합성과 갱신 전략이 빠진 답변.

### StampedLock → 정책 / 메뉴 캐시 갱신 중 조회 정합성

**질문**: "StampedLock을 쓴 이유와 커머스에서 같은 도구가 필요한 자리는?"

**1분 답변**:

> StampedLock을 고른 이유는 *읽기 압도적 + 갱신이 가끔* 패턴이었기 때문입니다. 정적 데이터 갱신 중 조회 요청이 일시적으로 NPE를 내던 문제였고, `writeLock`으로 갱신 구간을 보호하면서 조회는 `tryReadLock(2.5s)` 타임아웃을 둬서 *갱신 동안 무한 대기*도, *읽기 차단으로 인한 서비스 장애*도 막았습니다. 커머스에서 같은 자리는 운영자가 메뉴 가격·옵션·프로모션 정책을 *어드민에서 바꾸는 순간* 다중 서버 인메모리 캐시가 동시 갱신되는 구간입니다. 갱신은 짧지만 그동안 들어오는 주문 요청을 차단하면 안 되고, 갱신 중 정합성이 *조용히* 깨지면 매장 POS·결제 금액·쿠폰 계산까지 줄줄이 틀어집니다. 정확히 StampedLock 시나리오입니다.

**꼬리 질문 — "ReentrantReadWriteLock이 아닌 StampedLock을 굳이 쓴 이유?"**

> 읽기가 압도적인 환경에서 optimistic read를 쓸 여지가 있었고, 재진입이 필요한 자리가 아니었기 때문입니다. 락 자체 비용이 더 낮은 쪽을 골랐고, 대신 *재진입 금지 / Condition 없음*이라는 제약을 코드 컨벤션으로 명시해 동료가 잘못 쓰지 않게 했습니다.

**약한 답변**: "synchronized 쓰면 됩니다" — 읽기 차단 비용을 못 본 답변.

### AliasMethod O(1) → 쿠폰 / 추천 가중치 선택

**질문**: "AliasMethod 같은 알고리즘 결정을 커머스에서도 활용할 자리가 있나요?"

**1분 답변**:

> AliasMethod는 가중치 기반 랜덤 선택을 누적합 O(n)에서 사전 테이블 O(1)로 바꾼 결정입니다. JMH 기준으로 `ThreadLocalRandom` + AliasMethod가 `SecureRandom` 기반 누적합 대비 약 58배 빨랐고, 100만 스핀 시뮬레이션 시간이 의미 있게 줄었습니다. 커머스로 옮기면 *가중치 기반 추천 / A·B·C 군에 대한 트래픽 분배 / 가중치 있는 쿠폰 발급 / 매장 노출 우선순위 셔플*에 그대로 들어갑니다. 단, 가중치가 자주 바뀌면 테이블 재계산 비용이 생기므로 *변경 빈도 낮음 + 선택 빈도 높음* 조건일 때만 씁니다.

**꼬리 질문 — "SecureRandom을 왜 ThreadLocalRandom으로 바꿔도 된다고 판단했나요?"**

> 슬롯 서버 내부에서 결과를 만드는 데 *암호학적 강도*는 요구사항이 아니었습니다. 결과 위·변조를 막는 책임은 서버-클라이언트 프로토콜과 결과 영속화 쪽에 있었고, 랜덤 생성기 단계는 *고른 분포 + 락 경합 없음*이면 충분했습니다. JMH로 차이를 측정해 두고 결정 근거를 문서로 남겼습니다.

**약한 답변**: "빠른 알고리즘으로 바꿨습니다" — *왜 바꿔도 되는가*가 빠지면 보안 사고 답변이 된다.

### Kafka Outbox → 주문·결제 이벤트 신뢰성

**질문**: "Kafka로 후처리를 분리할 때 메시지 유실은 어떻게 막았습니까? 커머스 결제 도메인에 그대로 쓸 수 있나요?"

**1분 답변**:

> 핵심 결정은 두 가지입니다. 하나는 *언제 발행하는가* — `@TransactionalEventListener(AFTER_COMMIT)`로 *커밋 이후에만* 발행해 롤백된 트랜잭션 이벤트가 외부로 새는 것을 막았습니다. 다른 하나는 *실패한 메시지를 어디에 두는가* — 발행 실패 시 `Propagation.REQUIRES_NEW`로 원본 트랜잭션과 분리된 별도 트랜잭션에서 실패 메시지를 DB에 저장하고, traceId를 함께 적어 스케줄러가 재전송하면서 원인을 추적했습니다. 정식 Transactional Outbox와의 차이도 인지하고 있습니다 — AFTER_COMMIT과 Kafka 발행 사이 짧은 구간(JVM 크래시·SIGKILL)에 유실 가능성이 남습니다. 슬롯 도메인의 통계·알림은 이 수준이 허용 범위였지만, 커머스 *결제 승인 이벤트*는 비즈니스 데이터와 같은 트랜잭션에 outbox 테이블로 적재한 뒤 relay로 발행하는 *정식 Outbox* 구조로 가야 합니다. 비용은 outbox 테이블 폴링/CDC 한 단계 추가뿐이고, 결제 도메인의 정합성 비용 대비 충분히 쌉니다.

**꼬리 질문 — "정식 Outbox는 polling이 좋습니까, CDC가 좋습니까?"**

> 운영 인프라에 Debezium 같은 CDC 파이프라인이 이미 있으면 CDC, 없으면 단순 polling 한 테이블로 시작합니다. polling은 *지연 1\~3초*를 받아들이는 대신 운영 부담이 거의 없고, CDC는 지연이 짧은 대신 별도 컴포넌트의 가용성을 운영해야 합니다. 결제 같은 도메인은 polling 1초 + 멱등 소비로도 대부분 요구사항을 만족합니다.

**약한 답변**: "Kafka는 at-least-once니까 멱등 소비만 하면 됩니다" — *언제 발행하는가*와 *발행 실패를 어디에 두는가*가 빠진 답변.

### 다중 서버 캐시 정합성 → 메뉴 / 매장 / 쿠폰 캐시

**질문**: "어드민에서 메뉴 가격을 바꾸면 다중 서버 캐시가 어떻게 갱신되나요?"

**1분 답변**:

> 슬롯팀에서 했던 그대로 옮겨도 1차 구조가 됩니다. JPA 커밋 이벤트(`PostCommitUpdateEventListener`)로 *변경된 ID*만 RabbitMQ Fanout Exchange로 발행 → 각 서버가 자기 큐에서 수신 → 해당 항목만 선택 갱신 → `StampedLock` writeLock 구간으로 보호. 인스턴스가 수십\~수백 대 규모로 커지면 RabbitMQ Fanout 대신 Kafka 토픽에 *인스턴스마다 독립 consumer group*을 두거나 Redis Pub/Sub로 가는 것이 운영적으로 가볍습니다. 핫키에는 *probabilistic early expiration*과 single-flight를 두어 Cache Stampede를 막고, *진실 원천 = DB / L1 캐시 = Caffeine / 광역 캐시 = Redis*의 2-tier 구조로 피크 트래픽을 흡수합니다.

**꼬리 질문 — "변경 ID만 보내고 본문은 안 보내는 이유?"**

> 메시지 본문에 캐시 전체를 실으면 메시지가 크고, 발행과 수신 사이에 추가 변경이 발생할 경우 *오래된 본문*이 캐시로 들어가는 race가 생깁니다. 변경 ID만 보내고 각 서버가 *수신 시점의 DB*에서 본문을 다시 읽으면 race가 사라지고 메시지가 가벼워집니다.

**약한 답변**: "Redis로 캐시 공유하면 됩니다" — 어드민 변경 시 *원자성*과 *조회 차단 비용*이 빠진 답변.

## 상위 압박 질문 방어

### "TPS·레이턴시 숫자를 말씀해주세요"

> 죄송하지만 *정확한 TPS·레이턴시 수치*는 사내 자료에 해당해 외부에 말씀드릴 수 있는 형태로 정리해두지 않았습니다. 대신 측정 방식은 말씀드릴 수 있습니다. 슬롯 스핀 최적화는 JMH 마이크로벤치마크로 `ThreadLocalRandom` 70.241 ops/s vs `SecureRandom` 1.197 ops/s 약 58배 차이를 확인했고, 시뮬레이션 단위에서는 100만 스핀 기준 분 단위 경과 시간이 의미 있게 줄었습니다. 캐시 정합성 쪽은 *NPE 발생 빈도가 0*이라는 정성적 결과로 운영을 봤습니다. 입사 후 환경에 들어가면 *latency / error rate / saturation* 기준으로 처음부터 측정 가능한 형태로 정리하겠습니다.

### "MyBatis는 안 다뤄봤죠?"

> 솔직히 운영 단위로는 다루지 않았고, 학습 단위로만 만져봤습니다. 다만 SQL을 직접 작성·튜닝하는 감각 자체는 슬롯팀에서 캐시 충족 판정 쿼리를 *복합 인덱스* 단위로 EXPLAIN으로 읽으면서 튜닝한 경험으로 갖고 있고, 레거시 코드에서 XML mapper / 동적 SQL / resultMap을 빠르게 읽는 데 시간이 오래 걸리지 않을 자신이 있습니다. JPA-only 환경의 *N+1*과 MyBatis 환경의 *동적 SQL이 복잡해지는 양상*의 위험을 *같은 축*(SQL 가시성 vs 객체 매핑 편의)으로 이해하고 있어, 어떤 자리에 어떤 도구가 맞는지 빠르게 판단합니다.

### "K8s 직접 운영해보셨어요?"

> 직접 K8s 클러스터 *자체*를 운영하지는 않았습니다. NHN Cloud Container Service의 K8s 추상화 위에서 *사용자 관점*으로 다뤄봤고, 가장 길게 다룬 자리는 `terminationGracePeriodSeconds` 30초 고정 제약 하에서 preStop sleep 15초 + gRPC grace 12초 + 여유 3초로 *예산을 쪼개* 503을 제거한 경험입니다. Deployment·HPA·PDB 같은 객체는 *읽고 수정 PR을 내는* 수준까지였고, 직접 *튜닝 결정의 주인*은 아니었다는 점을 정직하게 말씀드립니다.

## 면접 직전 한 줄 체크리스트

답변이 길어진다 싶을 때 보는 짧은 점검표.

- 슬롯 단어를 한 번 말했다면 그 즉시 *커머스 단어*로 바꿔 한 번 더 말했는가
- 의사결정 *축* 6개 중 어디 위에 있는 답변인지 1초 안에 말할 수 있는가
- 숫자가 없으면 *측정 방법*과 *체감 단위*를 같이 말했는가
- "공부하겠습니다"로 끝나는 답변이 아닌가
- 슬롯/베팅 경험을 *깎으면서* 답하지 않았는가

## 자기 점검용 미니 시나리오

면접 1\~2일 전에 시간을 정하고 다음 시나리오를 큰 소리로 답하면서 60초 안에 끝나는지 확인한다.

```text
[시나리오 A] 자기소개 → 가장 어려웠던 동시성 문제 한 가지
[시나리오 B] 자기소개 → 슬롯과 커머스 도메인의 차이 → 본인이 가져갈 수 있는 의사결정 축
[시나리오 C] 자기소개 → Kafka Outbox 경험 → 결제 도메인에서 정식 Outbox로 가야 하는 이유
[시나리오 D] 자기소개 → 캐시 정합성 → 어드민 메뉴 변경 시 다중 서버 갱신
[시나리오 E] 자기소개 → RCC 백그라운드 캐시 → 피크 시간 매장 가용성 사전 적재
```

각 시나리오는 *답변 본문 60초 + 꼬리 질문 30초*까지 가는 것이 기본이다. 답이 길어지면 6축 중 어디로 돌아갈지 의식하면서 줄인다.

## 마무리

이 문서의 본질은 *슬롯 답변을 외우는 것*이 아니다. 자기 의사결정의 *축*을 알고, 그 축이 회사 도메인의 어느 자리에 들어가는지를 1분 안에 말할 수 있는 상태로 만드는 것이다. CJ푸드빌 디지털 채널이라는 환경에서 *주문·결제·쿠폰·매장 운영*이라는 도메인 단어가 처음일 수는 있어도, 그 안에서 풀게 될 문제의 *형태*는 이미 직접 풀어본 모양과 같다. 그 점을 면접 첫 5분 안에 보여주는 것이 본 답변집의 목표다.
