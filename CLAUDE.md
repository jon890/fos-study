## Repository Overview

개인 기술 블로그 겸 학습 기록 저장소. 마크다운 파일로 구성되며 GitHub 자동 sync를 통해 블로그에 게시된다. 빌드/테스트 명령어 없음 — 파일 작성과 구조 관리가 주 작업이다.

## 폴더 구조와 배치 기준

```

├── database/           # 데이터 스토어 전반
│   ├── mysql/
│   ├── opensearch/
│   └── redis/
├── java/               # Java 언어 및 Spring 생태계
│   ├── spring/

│   └── ...
├── devops/             # 인프라, 배포, 모니터링
│   ├── docker/
│   ├── k8s/

```

### 폴더 배치 결정 기준

- **database/** — 데이터를 저장·검색하는 스토어. MySQL, OpenSearch, Redis 포함. Kafka는 메시지 브로커라 최상위 유지
- **task/** — 실제 업무 구현 경험. 개념 설명 아닌 "내가 이걸 왜, 어떻게 만들었나" 기록
- **architecture/** — 특정 기술에 종속되지 않는 설계 개념 (전략 패턴 등). 실제 사례 링크는 포함하되 사례 본문은 `task/`에 작성
- 신규 기술 문서는 기존 최상위 폴더에 맞는 게 없을 때만 새 폴더 생성

## 문서 작성 규칙

### 개념 문서 vs 업무 기록 분리

- `architecture/`, `database/`, `java/` 등: 기술 개념 설명. 언제 읽어도 유효한 내용
- `task/<팀명>/`: 특정 시점의 구현 경험. 진행 기간 헤더 필수, 민감 정보 제거

### 민감 정보 제거 기준

업무 기록(`task/`) 작성 시:

- 회사명, 팀명, 내부 시스템 이름 → 일반 표현으로 대체 (예: Dooray → "사내 협업 도구")
- 내부 URL, IP → 제거 또는 `<내부-엔드포인트>` 로 대체
- 오픈소스 도구명(OpenSearch, Spring Boot 등)은 그대로 유지

### 코드 예시 검증

문서에 코드 블록을 포함할 때는 반드시 실제 코드에서 클래스명·메서드명·필드명을 확인 후 기재. 기억에 의존한 재현 금지.

### 마크다운 Bold + 괄호 패턴

`**텍스트(영문)**` 형태는 일부 파서에서 bold가 렌더링되지 않는다.
반드시 `**텍스트**(영문)` 형태로 작성한다.

- 올바른 예: `**구조**(harness)`, `**제어**(Control)`
- 잘못된 예: `**구조(harness)**`, `**제어(Control)**`

메서드명처럼 괄호가 이름의 일부인 경우(`**canRetry()**`)는 코드 인라인으로 표기한다: `` `canRetry()` ``

### 하위 문서 링크

관련 상세 문서가 저장소 내에 있으면 상대 경로로 링크. 링크 전 파일 존재 여부 확인.

### 가시성 원칙 — 스캔 가능한 형태로 쓴다

문서는 정보 밀도만큼이나 **시각적으로 빠르게 훑을 수 있는가**가 중요하다. 글로 다 풀어 쓰면 정보가 있어도 독자가 못 본다.

- **항목이 3개 이상이면 콤마 줄글이 아니라 리스트** — "A + B + C + D 시간 예산 설계로 ..."처럼 4개 컴포넌트를 콤마로 잇지 말고 bullet 4개로 분리
- **링크 제목은 짧게, 본문 설명은 다음 줄로 분리** — `[긴 제목 — 부연 설명](path) — 또 긴 줄글` 패턴 금지. 제목은 짧게, 설명은 다음 단락 또는 bullet으로
- **사례 누적이 예상되는 섹션은 H3 또는 표 구조로 시작** — 사례가 1건이어도 미리 `### 사례명` 구조로 잡아두면 누적 시 자연스럽게 확장된다
- **본문과 메타 설명 분리** — "이 글의 X 단락이 정확히 그 환경" 같은 cross-reference 메모는 본문 줄에 묻지 말고 별도 줄로 띄움
- **단락 안에 정보가 4개 이상 묶여 있으면 분해** — 한 paragraph가 정보 항목을 4개 이상 안고 있으면 사람 눈이 흐른다. 헤더·리스트·표 중 하나로 쪼갠다

### 각 팀 README.md 구성 규칙

- 상단에 **회사·팀·기간·기술 스택** 메타 정보
- **문서 목록 테이블** — 카테고리로 구분: `신규 구현` / `트러블슈팅` / `리팩터링` / `아키텍처 개선`. 카테고리는 강제 아니고 해당 팀에 의미 있는 것만.
- 각 행: `기간 | 업무 한 줄 요약 | 문서 링크`
- 하단에 **기술 키워드** 태그 (선택)

### 문서 작성 규칙

- 진행 기간 헤더 필수 (`**진행 기간**: 2024.06 ~ 2024.12`)
- 회사명·팀명·내부 시스템 이름은 일반 표현으로 대체. 단, README의 메타 정보(회사/팀)는 예외로 표기.
- 내부 운영 중인 상품·서비스명은 공개하지 않고 메커닉/구조 조합으로 일반화 (예: nsc-slot 팀의 슬롯 상품명)
- 새 문서 추가 시 해당 팀 `README.md`의 문서 목록 테이블도 함께 업데이트

## Git 커밋 규칙

커밋은 항상 **관심사 단위로 나눠서 원자적으로** 만든다. 한 커밋은 하나의 논리적 변경만 담는다.

- 파일 폐기 / 신규 산출물 추가 / 스킬·설정 강화가 한 작업에서 동시에 발생해도 **각각 별도 커밋**으로 분리한다
- 사용자가 "여기까지 커밋 푸시" 라고 해도 자동으로 한 커밋에 합치지 않는다. 변경 묶음을 먼저 식별하고 관심사별로 stage → commit 을 반복한다
- 예: 이력서 drop + 신규 이력서 추가 + 스킬 강화 → 3개 커밋 (각각 `docs(resume): drop ...`, `docs(resume): add ...`, `docs(skill): strengthen ...`)
- 예외: 같은 관심사의 자명한 부산물(HTML + 그로부터 생성된 PDF, 새 문서 + 해당 README 인덱스 갱신)은 한 커밋에 묶어도 된다
