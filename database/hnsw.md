# HNSW 깊이 보기 — 벡터 검색은 어떻게 빠른가

벡터 DB 가 수천만 개의 벡터 속에서 밀리초 안에 "가장 비슷한 것"을 찾아내는 비결은 ANN(근사 최근접) 인덱스다.
그중 사실상 업계 표준이 된 게 HNSW 다. OpenSearch · Milvus · Qdrant · pgvector 까지 거의 다 이걸 기본으로 쓴다.
이 글은 HNSW 가 어떤 아이디어 위에 서 있고, 색인·검색이 어떻게 동작하며, 메모리를 왜 그렇게 먹는지 정리한 기록이다.

---

## 왜 "근사"인가

가장 비슷한 벡터를 정확히 찾으려면 쿼리와 모든 벡터의 거리를 다 계산하면 된다(brute-force, FLAT 인덱스).
문제는 비용이다. 1,600만 벡터 × 1024차원이면 쿼리 한 번에 1,600만 번의 거리 계산을 해야 한다. 정확하지만 느리다.

그래서 **약간의 정확도를 양보하고 속도를 얻는** 게 ANN(Approximate Nearest Neighbor)이다.
"진짜 1등을 가끔 놓치더라도 거의 항상 상위권은 맞추는" 전략이다. 이 정확도를 [recall](./vectordb-comparison.md) 로 측정한다.

---

## HNSW 의 두 뿌리

HNSW 는 **H**ierarchical **N**avigable **S**mall **W**orld 의 약자다. 이름에 두 아이디어가 다 들어 있다.

### Navigable Small World (작은 세상 그래프)

노드(벡터)들을 가까운 것끼리 이어 그래프로 만든다. 검색은 아무 노드에서 출발해, 이웃 중 쿼리에 더 가까운 쪽으로 계속 이동하는 **greedy 탐색**을 하다가 더 못 가까워지면 멈춘다.
"여섯 다리만 건너면 누구와도 연결된다"는 작은 세상 이론처럼, 잘 만든 그래프는 몇 홉만에 목적지 근처에 도달한다.

문제는 그래프가 한 층뿐이면 출발점이 멀 때 잔걸음이 많다는 것이다.

### Hierarchical (계층 = skip list 아이디어)

그래서 층을 쌓는다. 링크드 리스트에 고속 레인을 얹어 탐색을 빠르게 하는 skip list 처럼, HNSW 는 그래프를 여러 층으로 만든다.

```
Layer 2 (희소)  : ●───────────────●         long-range, 고속도로
Layer 1 (중간)  : ●─────●─────●───●
Layer 0 (전체)  : ●─●─●─●─●─●─●─●─●         모든 노드, 골목길
```

위층은 노드가 적고 멀리 연결돼 빠르게 근처까지 데려다주고, 아래로 갈수록 촘촘해져 정밀하게 좁힌다.

---

## 색인 — 노드를 어떻게 넣나

새 벡터가 들어오면:

1. **레벨을 무작위로 정한다.** 지수 분포라 대부분은 layer 0 에만, 소수만 위층까지 올라간다. 그래서 위층은 자연히 희소해진다.
2. 최상위 층부터 greedy 로 진입점을 찾아 내려온다.
3. 자기 레벨의 각 층에서 `efConstruction` 만큼 후보를 탐색해, 그중 **가까운 이웃 M 개와 연결**한다. layer 0 에서는 보통 2M 개까지 잇는다.

`M` 이 그래프의 밀도(노드당 링크 수), `efConstruction` 이 빌드 품질을 결정한다.

---

## 검색 — 어떻게 빠르게 찾나

1. 최상위 층의 진입점에서 시작한다.
2. 현재 노드의 이웃 중 쿼리에 더 가까운 노드로 이동한다. 더 못 가까워지면 한 층 내려간다.
3. 최하층(layer 0)에서 `ef` 개의 후보를 우선순위 큐로 정밀 탐색한다.
4. 그중 상위 k 개를 반환한다.

지도에서 고속도로로 대충 근처까지 간 뒤 국도 → 골목으로 좁히는 것과 같다. 전수 비교 대신 그래프를 따라 수백 번만 점프해서 근사 최근접을 찾으니 빠르다.

---

## 핵심 파라미터

| 파라미터 | 의미 | 올리면 |
| --- | --- | --- |
| `M` | 노드당 이웃 링크 수 | recall ↑, 메모리 ↑ |
| `efConstruction` | 색인 시 탐색 후보 수 | 그래프 품질 ↑, 빌드 느림 |
| `ef`(efSearch) | 검색 시 탐색 후보 수 | recall ↑, 검색 느림 |

특히 `ef` 는 **런타임에 조절**할 수 있다. 같은 인덱스로도 "빠르게 대충"과 "느리게 정확히"를 쿼리마다 바꿀 수 있는 게 실무에서 큰 장점이다.

---

## 메모리에 무엇이 올라가나

HNSW 가 빠른 대가는 메모리다. 세 가지가 올라간다.

1. **벡터 원본** — 거리 계산에 실제 값이 필요해서, 양자화를 안 하면 full-precision 이 그대로 메모리에 있어야 한다. 1024차원 × 4byte = 노드당 4KB. **이게 메모리의 대부분이다.** 1,600만 개면 raw 약 68GB.
2. **그래프 링크** — 각 노드가 든 이웃 id 리스트. 노드당 최대 M(또는 layer 0 의 2M)개 × 레이어 수. 벡터보다는 작지만 무시할 수 없다.
3. **id 매핑** — 내부 id 와 외부 문서 id 의 대응.

메모리가 부담되면 줄이는 길이 있다.

- **양자화** — PQ/SQ 로 벡터를 압축한다. float32 → int8 이면 약 1/4. recall 을 조금 내주고 메모리를 크게 아낀다.
- **DiskANN/mmap** — 그래프·벡터를 디스크에 두고 일부만 메모리에 올린다. 검색 지연이 늘지만, 트래픽이 낮으면 충분히 감당된다.

## 장단점과 대안

**장점**

- 빠르고 recall 이 높다(잘 튜닝하면 0.95 이상).
- 데이터를 하나씩 동적으로 추가할 수 있다(IVF 처럼 전체 재학습이 필요 없다).
- `ef` 로 속도와 정확도를 런타임에 조절한다.

**단점**

- 메모리를 많이 쓴다(벡터를 다 올리면).
- 삭제가 비효율적이다(그래프에서 노드를 빼기 어려워, 보통 표시만 하고 주기적으로 재구성).
- 빌드가 IVF 보다 느리다.

**언제 다른 걸 쓰나**

- 메모리가 빠듯한 초대용량 → DiskANN(온디스크) 또는 양자화.
- 데이터가 정적이고 메모리 절약이 우선 → IVF 계열.
- 대부분의 일반적인 경우 → 그냥 HNSW 가 무난한 기본값이다.

---

## 정리

- HNSW = 작은 세상 그래프 + 계층(skip list) 으로, 전수 비교 없이 근사 최근접을 빠르게 찾는다.
- 검색은 위층에서 대충, 아래층에서 정밀하게 좁히는 greedy 탐색이다.
- `M`·`efConstruction` 은 빌드, `ef` 는 검색 품질을 정하고, `ef` 는 런타임 조절이 된다.
- 빠른 대가로 벡터를 메모리에 올려야 하지만, 양자화·DiskANN 으로 줄일 수 있다.

벡터 DB 별로 HNSW 가 어떻게 쓰이는지는 [벡터 DB 선택 가이드](./vectordb-comparison.md)와 [OpenSearch vs Milvus](./milvus/opensearch-vs-milvus.md)에 정리해 두었다.
