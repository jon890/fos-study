## 업무 기록 문서 작성 가이드

새 업무 기록 문서를 작성하거나 기존 문서를 수정할 때 아래 규칙을 따른다.

### 톤과 문체

- **한국어**, 1인칭 경험 서술체 ("~했다", "~이었다", "~이다")
- 기술 블로그 스타일 — 독자가 실무자라고 가정하고 배경 설명은 간결하게
- "왜 이 선택을 했는가"를 중심으로 서술. 단순한 구현 나열보다 의사결정 배경 우선
- 문제를 먼저 제시하고 해결 방향을 전개하는 구조 선호

### 문서 구조

1. **어떤 서비스/기능인가** — 한두 문단으로 맥락 설명. 독자가 이 기능이 왜 필요한지 이해할 수 있어야 함
2. **왜 이 기술/설계를 선택했나** — 선택지와 트레이드오프. 대안이 있었다면 왜 채택하지 않았는지 포함
3. **설계 및 구현 상세** — 핵심 코드 발췌 포함. 전체 코드보다 의도를 드러내는 부분만
4. **트러블슈팅** (해당하는 경우) — 문제 상황 → 원인 → 해결 흐름으로 서술
5. **마무리** — 이 과정에서 배운 점, 다음에 다르게 할 것

### 코드 예시 규칙

- 핵심 의도가 드러나는 부분만 발췌. 전체 클래스/메서드 불필요
- `// ❌ before` / 변경 후 패턴으로 개선 전후를 대비해서 보여주는 방식 선호
- 코드 블록에 언어 명시 (`java`, `json`, `yaml` 등)

### README 업데이트

새 문서를 추가하면 해당 팀 디렉터리의 `README.md`도 함께 업데이트한다.
- 문서 목록 테이블에 행 추가
- 기간별 주요 업무 섹션에 요약 추가
- 필요하면 기술 키워드 보강
