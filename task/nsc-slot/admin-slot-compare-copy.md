# Admin 슬롯 비교/복사 기능 개발

**진행 기간**: 2024.07 ~ 2024.12

---

## 배경

신규 슬롯을 개발하면 Alpha 환경에서 검증 후 Real 환경에 배포한다. 이 과정에서 두 환경의 슬롯 설정 데이터가 일치하는지 확인하고, Alpha 데이터를 Real로 복사하는 작업이 필요했다.

기존에는 수동으로 했는데, 슬롯 설정 구조가 복잡하고 실수가 생기기 쉬워서 어드민 UI에 비교/복사 기능을 추가하게 됐다.

---

## 기능 개요

```
Alpha DB                    Real DB
  슬롯 설정 ──── 비교 ────▶  슬롯 설정
                 ↓
           다른 항목 목록 반환

Alpha DB                    Real DB
  슬롯 설정 ──── 복사 ────▶  슬롯 설정 (덮어쓰기)
                              단, version/uuid는 새로 부여
```

---

## 구현에서 어려웠던 것들

### 깊은 비교 (Deep Comparison)

슬롯 설정은 중첩된 객체 구조다. 단순 `equals`로는 비교가 안 됐다.

특히 두 가지 케이스가 까다로웠다.

**record class의 배열 프로퍼티**

Java record는 기본적으로 `equals`를 자동 생성하지만, 배열 타입 프로퍼티는 참조 비교를 하기 때문에 내용이 같아도 다르다고 판정했다.

```java
// record의 기본 equals: 배열은 참조 비교
record SlotConfig(int[] payouts) {} // payouts.equals()는 참조 비교

// 해결: 배열 프로퍼티가 있는 record는 equals 오버라이딩
```

record 클래스에 배열 프로퍼티가 있으면 `Arrays.equals`를 사용하도록 오버라이딩 처리를 추가했다.

**비교 대상이 null인 경우**

Real DB에 아직 없는 항목을 비교할 때, target 객체가 null이면 예외가 났다. null이면 비교하지 않고 바로 "다르다"고 판정하도록 처리했다.

**잭팟 비교**

잭팟 설정은 UUID를 포함하는데, 환경마다 UUID가 달라서 단순 비교 시 항상 "다르다"고 나왔다. UUID를 비교 대상에서 제외하고, 잭팟 설정의 실제 내용(금액, 조건 등)만 비교하도록 했다.

### 복사 시 고유값 재생성

Alpha 데이터를 Real로 복사할 때 그대로 복사하면 안 되는 값들이 있다.

| 항목 | 이유 |
|------|------|
| `version` | 환경마다 독립적으로 관리 |
| 잭팟 `uuid` | Real에서 새 잭팟 객체로 생성해야 함 |

복사 로직에서 이 값들을 새로 부여하도록 처리했다. 잭팟의 경우 Real에 이미 존재하면 내용만 복사하고, 없으면 새로 생성하도록 분기했다.

### 비교 DB 환경 설정

비교 대상 DB가 바뀌면서 설정을 여러 번 수정했다. 처음에는 로컬 dev DB에서 비교하다가, AWS Alpha로 변경하면서 환경 설정을 프로퍼티화했다.

### 어드민 슬로우 쿼리

슬롯 목록 조회 쿼리가 느렸다. 슬롯이 늘어날수록 응답 시간이 길어졌다.

쿼리 실행 계획을 확인해서 불필요한 JOIN을 제거하고, 내부 로직에서 N+1 문제도 해결했다.

---

## 배운 것

**깊은 객체 비교는 생각보다 엣지 케이스가 많다.** record의 배열 프로퍼티, null 처리, 고유값 제외 등 실제로 써봐야 보이는 케이스들이 있었다. 복잡한 도메인 객체의 비교 로직은 단위 테스트를 꼼꼼하게 작성해야 한다.

**복사 기능은 멱등성을 고려해야 한다.** 같은 데이터를 여러 번 복사해도 결과가 동일해야 한다. 잭팟 복사 로직(없으면 생성, 있으면 업데이트)을 이 관점에서 설계했다.
